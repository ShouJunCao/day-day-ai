2.1 OpenAI 兼容 API 标准

Chat Completions · Streaming · Function Calling · Responses API — 一套标准，万物互联

💡 核心洞察

OpenAI 的 API 格式已经成为大模型行业的事实标准。通义千问、DeepSeek、智谱、Moonshot、Mistral 等国内外主流模型全部兼容这套接口。掌握一套 API，就能调用所有模型——这就是"OpenAI 兼容"的力量。

一、为什么需要统一标准？

在 2023 年之前，每家大模型厂商都有自己的 API 格式：调用通义千问用一种方式，调用文心一言用另一种方式，调用智谱 GLM 又是不同的方式。开发者如果想切换模型或做多模型对比，就得写完全不同的对接代码。这种碎片化严重拖慢了开发效率。

OpenAI 的 Chat Completions API 凭借其简洁的设计和强大的功能，逐渐成为行业标杆。2024 年以来，几乎所有新发布的大模型都宣称"兼容 OpenAI API 格式"。这意味着你只要学会了 OpenAI 的 openai Python 库，就能用同一套代码调用市面上绝大多数模型。

这种"一次学习，到处运行"的体验，就像 SQL 之于数据库、HTTP 之于 Web——标准统一带来了巨大的生态红利。开发者不再被单一厂商绑定，可以自由切换模型、做 A/B 测试、构建多模型网关，灵活性大大增强。

从技术架构的角度看，OpenAI 兼容 API 遵循 RESTful 设计原则：使用标准 HTTP 方法（POST）、JSON 格式的请求和响应体、以及明确的 HTTP 状态码。认证方式统一采用 Authorization: Bearer <api_key> 的 HTTP Header 方式。这种简洁的设计降低了接入门槛，让任何支持 HTTP 的编程语言都能轻松对接。

Python 开发者最常用的工具是官方的 openai 库。这个库封装了所有 API 调用的细节，提供了类型安全的接口和完善的错误处理。安装只需 pip install openai，然后创建 OpenAI 客户端实例即可开始调用。值得注意的是，国内模型同样支持这个库——只需修改 base_url 参数指向自己的 API 地址。

二、Chat Completions API — 对话的核心

💬 Chat Completions API

基础接口所有模型支持RESTful

Chat Completions 是大模型对话的核心接口。你发送一组消息（messages），模型返回一个完整的回复。接口地址固定为 /v1/chat/completions。

一个标准的 Chat Completions 请求包含以下核心元素：

参数类型说明示例
modelstring模型名称qwen-plus, gpt-4o
messagesarray消息列表（核心）包含 system/user/assistant 消息
temperaturefloat随机性控制（0~2）0.7（平衡创造性和准确性）
max_tokensint最大输出 token 数500
streambool是否流式输出false（默认）

消息（messages）是 Chat Completions 的灵魂。每条消息有一个 role（角色）和 content（内容）：

角色用途使用场景
system设定 AI 的行为准则和角色定位对话最前面，定义 AI 的身份和规则
user用户的输入每轮用户说的话
assistantAI 的历史回复多轮对话时，记录 AI 说过的话
tool工具调用的返回结果Function Calling 中使用

三、流式输出（Streaming）— SSE 的艺术

🌊 流式输出（Streaming）

SSE 协议逐 token 返回用户体验关键

非流式模式下，模型生成完整回复后一次性返回。用户需要等待数秒甚至十几秒才能看到结果。流式模式让模型边生成边返回，用户几乎实时看到文字"打"出来——就像 ChatGPT 那样。

流式输出的底层技术是 SSE（Server-Sent Events）。SSE 是一种基于 HTTP 的单向通信协议：服务器通过一个持久连接，持续向客户端推送事件。每个事件以 data: 开头，以 \n\n 结尾。

在 Python 的 openai 库中，流式模式的用法非常简洁。设置 stream=True 后，API 返回的不是一个完整的 Response 对象，而是一个迭代器。每个迭代元素是一个 chunk（数据块），其中包含一个 delta（增量），代表本次新生成的那部分文本。

# 流式输出示例 — 核心代码只需 3 步

# 1. 开启 stream=True
stream = client.chat.completions.create(
    model="qwen-plus",
    messages=[{"role": "user", "content": "介绍 GIL"}],
    stream=True,  # 开启流式
)

# 2. 逐块读取增量
for chunk in stream:
    delta = chunk.choices[0].delta
    if delta.content:
        print(delta.content, end="", flush=True)

    # 3. 检查是否结束
    if chunk.choices[0].finish_reason:
        print(f"\n完成: {chunk.choices[0].finish_reason}")

流式与非流式的核心区别在于响应的数据结构不同。非流式返回完整的 message（包含完整 content），流式返回 delta（只包含本次增量）。此外，流式模式下的 usage（token 统计）信息通常在最后一个 chunk 中返回，有些提供商可能不返回。

在生产环境中，流式输出有几个需要注意的要点：错误处理——流式过程中网络中断可能导致数据不完整，需要实现超时和重试机制；背压控制——如果客户端处理速度跟不上生成速度，缓冲区可能溢出；结束标记——OpenAI 标准的流式响应以 data: [DONE] 标记结束，Python 的 openai 库会自动处理这个标记。

流式输出还有一个重要的应用场景：长文本生成的用户体验优化。当模型需要生成几千字的长文时，非流式模式可能让用户等待 10-30 秒才看到结果，而流式模式让用户在第一个 token 生成时（通常 <1 秒）就能看到内容。这种"即时反馈"对用户体验的提升是显著的。

四、Function Calling — 让模型调用外部工具

🔧 Function Calling（函数调用）

高级功能工具集成Agent 基础

Function Calling 让模型不再只是"回答问题"，而是能调用外部函数来获取实时信息、执行操作。这是构建 Agent 系统的基石。

Function Calling 的核心思想是：你定义一组工具描述（JSON Schema 格式），告诉模型"你可以调用这些函数"。模型在收到用户请求后，自主判断是否需要调用工具，如果需要，它会返回一个工具调用请求（包含函数名和参数），而不是直接生成文本回复。

完整的 Function Calling 流程分四步：

步骤角色做什么
❶ 定义工具开发者用 JSON Schema 描述函数名、参数、用途
❷ 模型决策AI 模型判断是否需要调用工具，返回 tool_calls
❸ 执行函数开发者调用实际函数，获取结果
❹ 返回结果AI 模型根据函数结果生成最终回复

图 3：Function Calling 四步流程

Function Calling 的关键代码是工具定义。工具描述使用 JSON Schema 格式，需要指定函数名（name）、功能描述（description）和参数结构（parameters）。参数描述越清晰，模型的调用准确性越高。

关于 Function Calling 有几个重要的实践要点：

• tool_choice 参数：设为 "auto"（默认）时模型自行决定是否调用；设为 "required" 时强制调用；设为 "none" 时禁止调用。还可以指定具体函数名强制调用特定工具。
• 并行调用：部分模型（如 GPT-4o、Qwen-Max）支持在一次响应中返回多个 tool_calls，开发者需要逐个执行并返回结果。
• 安全性：模型只返回"要调用什么"，实际执行由开发者控制。务必对模型传入的参数做校验和过滤，防止注入攻击。
• 错误处理：函数执行失败时，应将错误信息作为 tool message 返回给模型，让模型理解失败原因并尝试其他方式。

# Function Calling 工具定义示例

tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "获取指定城市的当前天气信息",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "城市名称，如：北京、上海",
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                },
            },
            "required": ["city"],
        },
    },
}]

# 发送请求时带上 tools
response = client.chat.completions.create(
    model="qwen-plus",
    messages=[{"role": "user", "content": "北京天气怎么样？"}],
    tools=tools,
    tool_choice="auto",  # auto=模型自行决定是否调用
)

五、Responses API — 新一代接口

🚀 Responses API

最新标准2025 年推出内置工具

Responses API 是 OpenAI 在 2025 年推出的新一代接口标准，旨在统一对话、工具调用和内置能力。接口地址为 /v1/responses，用更简洁的方式实现 Chat Completions + Function Calling 的全部功能。

Responses API 相比 Chat Completions 有三个核心改进：

• 统一的消息模型：不再区分 messages 和 tool_calls，所有输入输出都是统一的 input/output 格式
• 内置工具：原生支持 web_search（网页搜索）、file_search（文件检索）、code_interpreter（代码执行）等内置工具，无需自己实现
• 多步推理：原生支持模型在一次请求中多次调用工具、多步推理后给出最终答案

不过需要注意的是，Responses API 目前只有 OpenAI 官方支持，国内模型尚未跟进。在实际项目中，Chat Completions 仍然是最通用的选择。了解 Responses API 有助于把握未来趋势。

六、多轮对话与上下文管理

Chat Completions API 本身是无状态的——每次请求都是独立的，模型不会自动记住上一次对话的内容。要实现多轮对话，开发者需要在每次请求中携带完整的消息历史（messages 列表）。

这意味着每轮对话都需要把之前的所有消息（system + 历次 user/assistant 交互）一起发送给模型。随着对话轮次增加，消息列表会越来越长，token 消耗也会线性增长。当消息总长度接近模型的上下文窗口限制时，就需要进行上下文管理。

策略做法优点缺点
全量保留所有消息原样传递信息完整无丢失token 消耗大，可能超限
滑动窗口只保留最近 N 轮对话简单可控早期信息丢失
摘要压缩用 LLM 把旧消息压缩成摘要保留关键信息额外调用成本
Token 预算按 token 预算裁剪消息精确控制实现复杂

在实际项目中，推荐的做法是：保留 system 消息（始终在最前面）+ 最近 5~10 轮对话。如果对话特别长，可以用一个小模型（如 qwen-flash）对更早的对话做摘要压缩，将摘要作为一条 system 消息插入。这种混合策略在保证上下文完整性的同时，有效控制了 token 消耗。

七、模型切换：一套代码调用所有模型

OpenAI 兼容 API 的最大价值在于模型无关性。同一套代码，只需修改 base_url 和 api_key，就能切换到不同的模型：

模型base_url默认模型名
通义千问dashscope.aliyuncs.com/compatible-mode/v1qwen-plus
DeepSeekapi.deepseek.com/v1deepseek-chat
智谱 GLMopen.bigmodel.cn/api/paas/v4glm-4
Moonshotapi.moonshot.cn/v1moonshot-v1-8k
OpenAIapi.openai.com/v1gpt-4o

这种设计模式通常被称为"适配器模式"。在实际项目中，建议封装一个统一的 LLMClient 类，内部根据配置切换到不同的 base_url。这样业务代码完全不感知底层模型的变化，切换模型只需改一行配置。

八、总结与实践建议

📋 核心要点回顾

1. OpenAI API 是事实标准。几乎所有主流大模型都兼容 Chat Completions 接口格式。

2. 四个核心能力。Chat Completions（对话）、Streaming（流式）、Function Calling（工具调用）、Responses API（新一代标准）。

3. 流式输出是用户体验关键。设置 stream=True，逐 chunk 读取 delta 增量，实时展示生成过程。

4. Function Calling 是 Agent 基石。模型决定调用 → 开发者执行 → 结果返回 → 模型生成回复。

5. 一套代码切换模型。只改 base_url + api_key，即可无缝切换通义千问、DeepSeek、GLM 等模型。

掌握了 OpenAI 兼容 API 标准，你就拥有了调用几乎所有大模型的"万能钥匙"。下一节我们将深入国内各主流模型的 API 接入细节，了解每家在兼容标准之外的独特能力和注意事项。
