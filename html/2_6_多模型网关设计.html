2.6 多模型网关设计

📋 本节导读

在实际生产环境中，我们通常需要接入多个大模型提供商（OpenAI、DeepSeek、通义千问等），并且需要实现智能路由、故障转移、预算控制等功能。多模型网关就是为了解决这些问题而设计的统一中间层。

一、为什么需要多模型网关？

1.1 单一模型的局限性

直接使用单一模型提供商存在以下问题：

  • 单点故障：如果该模型服务不可用，整个应用瘫痪
  • 成本不透明：无法在不同价格模型间智能切换
  • API差异：不同提供商的API格式可能不同
  • 缺乏控制：难以实现统一的预算限制和访问控制

1.2 网关的核心价值

多模型网关作为应用的中间层，提供以下能力：

能力 说明 业务价值
统一接口 所有模型使用相同的OpenAI兼容API格式 应用代码无需修改即可切换模型
智能路由 根据成本、质量、可用性自动选择模型 优化成本和性能
故障转移 模型故障时自动切换到备选模型 提高系统可靠性
预算控制 全局和用户级别的费用限制 防止费用失控
监控日志 记录所有请求、响应、费用 便于问题排查和成本分析
💡 核心洞察：多模型网关本质上是一个API代理层，它将复杂的模型管理逻辑封装在内部，对外提供简洁统一的接口。这种设计模式在微服务架构中非常常见。

三、核心功能实现

3.1 配置管理（config.py）

使用Pydantic进行类型安全的配置管理，支持YAML文件和环境变量：

# 配置模型定义
class ModelConfig(BaseModel):
    name: str
    provider: str
    api_key: str
    base_url: str
    input_price: float  # 每百万token
    output_price: float
    enabled: bool = True

class GatewayConfig(BaseModel):
    gateway: GatewaySettings
    models: List[ModelConfig]
    routing: RoutingConfig
    budget: BudgetConfig
    
    @classmethod
    def from_yaml(cls, path: str) -> "GatewayConfig":
        """从YAML文件加载配置，支持环境变量替换"""
        with open(path, 'r', encoding='utf-8') as f:
            raw = yaml.safe_load(f)
        raw = cls._resolve_env_vars(raw)  # 替换 ${VAR} 格式
        return cls(**raw)
✨ 设计亮点：使用${ENV_VAR}语法引用环境变量，避免将API Key硬编码。例如：api_key: "${OPENAI_API_KEY}"

3.2 模型注册表（registry.py）

管理所有模型的客户端实例和健康状态：

class ModelRegistry:
    """模型注册表 - 管理所有可用模型"""
    
    def __init__(self, model_configs: List[ModelConfig]):
        self.configs: Dict[str, ModelConfig] = {}
        self.clients: Dict[str, AsyncOpenAI] = {}
        self.health_status: Dict[str, bool] = {}
        
        for config in model_configs:
            if config.enabled:
                self.register(config)
    
    def register(self, config: ModelConfig):
        """注册模型并创建客户端"""
        self.configs[config.name] = config
        self.clients[config.name] = AsyncOpenAI(
            api_key=config.api_key,
            base_url=config.base_url,
            timeout=float(config.timeout),
        )
        self.health_status[config.name] = True
    
    def mark_unhealthy(self, model_name: str):
        """标记模型为不健康（触发故障转移）"""
        self.health_status[model_name] = False

3.3 智能路由（router.py）

支持多种路由策略，自动选择最佳模型：

class ModelRouter:
    """智能路由器 - 根据策略选择模型"""
    
    def select_model(self, preferred: str = None, 
                     max_cost: float = None) -> str:
        """选择最佳模型
        
        策略：cost_first | quality_first | round_robin
        """
        if preferred and self._is_available(preferred, max_cost):
            return preferred
        
        available = self._get_available_models(max_cost)
        if not available:
            raise ValueError("没有可用的模型")
        
        if self.config.strategy == "cost_first":
            return self._select_by_cost(available)
        elif self.config.strategy == "quality_first":
            return self._select_by_quality(available)
        else:
            return self._select_round_robin(available)
    
    def get_fallback_model(self, failed: str) -> str:
        """获取故障转移模型"""
        for fallback in self.config.fallback_models:
            if fallback != failed and self._is_available(fallback):
                return fallback
        return None
路由策略 选择逻辑 适用场景
cost_first 按input_price升序，选最便宜的 日常对话、简单任务
quality_first 按input_price降序，选最贵的 复杂推理、专业任务
round_robin 轮询选择所有可用模型 负载均衡、测试

3.4 中间件系统（middleware.py）

使用FastAPI中间件实现横切关注点：

# 1. 请求日志中间件
class LoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.time()
        response = await call_next(request)
        duration = time.time() - start
        logger.info(f"{request.method} {request.url.path} "
                   f"- {response.status_code} - {duration:.3f}s")
        return response

# 2. 速率限制中间件（滑动窗口算法）
class RateLimitMiddleware(BaseHTTPMiddleware):
    def __init__(self, app, rate_limit: int):
        super().__init__(app)
        self.rate_limit = rate_limit
        self.timestamps: Dict[str, list] = defaultdict(list)
    
    async def dispatch(self, request, call_next):
        user_id = request.client.host
        if not self._check_rate_limit(user_id):
            return Response(status_code=429, 
                          content="Rate limit exceeded")
        return await call_next(request)

# 3. 预算控制中间件
class BudgetMiddleware(BaseHTTPMiddleware):
    def record_cost(self, user_id: str, cost: float):
        self.global_daily_cost += cost
        self.user_daily_costs[user_id] += cost
    
    def check_budget(self, user_id: str, estimated: float) -> bool:
        if self.global_daily_cost + estimated > self.config.global_daily_limit:
            return False
        if self.user_daily_costs[user_id] + estimated > self.config.user_daily_limit:
            return False
        return True
⚠️ 注意：中间件执行顺序很重要。FastAPI中先添加的后执行（洋葱模型）。建议顺序：日志 → 限流 → 预算控制。

3.5 故障转移实战

故障转移是网关最重要的可靠性保障。当模型调用失败时，网关会自动尝试备选模型：

# main.py 中的故障转移逻辑
@app.post("/v1/chat/completions")
async def chat_completions(request: ChatRequest, http_request: Request):
    user_id = http_request.client.host if http_request.client else "unknown"
    
    # 1. 选择模型
    selected_model = router.select_model(preferred_model=request.model)
    client = registry.get_client(selected_model)
    
    # 2. 调用模型API（带故障转移）
    try:
        response = await client.chat.completions.create(**kwargs)
    except Exception as e:
        logger.error(f"模型 {selected_model} 调用失败: {e}")
        
        # 标记模型为不健康
        registry.mark_unhealthy(selected_model)
        
        # 尝试故障转移到备选模型
        fallback = router.get_fallback_model(selected_model)
        if fallback:
            logger.info(f"故障转移: {selected_model} -> {fallback}")
            fallback_client = registry.get_client(fallback)
            kwargs["model"] = fallback
            response = await fallback_client.chat.completions.create(**kwargs)
            selected_model = fallback
        else:
            raise HTTPException(status_code=503, detail="所有模型不可用")

故障转移的流程如下：

步骤 动作 说明
1 调用首选模型 根据路由策略选择的模型
2 捕获异常 网络超时、API错误等
3 标记不健康 后续请求不再路由到该模型
4 选择备选模型 从fallback_models列表中选择
5 重试请求 使用备选模型重新发起请求
6 返回响应 对客户端透明，感知不到切换
💡 生产建议：故障转移时应该记录日志并发送告警通知。如果频繁触发故障转移，说明某个模型可能存在系统性问题，需要运维介入排查。
