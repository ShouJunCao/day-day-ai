2.7 API重试、限流与错误处理最佳实践

📋 本节导读

在生产环境中，API调用不可避免地会遇到各种错误：网络超时、服务不可用、请求限流等。如何优雅地处理这些错误，保证系统的稳定性和可靠性，是每个开发者必须掌握的技能。本节将系统介绍重试策略、限流器、熔断器等核心模式，并提供完整的生产级实现。

一、为什么需要错误处理？

1.1 常见的API错误类型

错误类型 HTTP状态码 原因 处理策略
网络超时 408, 504 网络连接不稳定、服务器响应慢 重试（指数退避）
服务器错误 500, 502, 503 服务器内部故障、过载 重试 + 熔断
请求限流 429 请求频率超过API限制 等待 + 限流器
认证失败 401, 403 API Key无效、权限不足 不重试（更新凭证）
参数错误 400 请求参数无效 不重试（修正参数）

1.2 错误处理的核心原则

💡 核心原则：
  • 分类处理：不同类型的错误需要不同的处理策略
  • 可重试性：只对瞬态错误重试，不对永久性错误重试
  • 退避策略：使用指数退避避免过载
  • 熔断保护：防止故障服务拖垮整个系统
  • 优雅降级：提供备选方案，保证核心功能可用

二、重试策略（Retry）

2.1 指数退避重试

指数退避是最常用的重试策略，每次重试的等待时间呈指数增长：

# 指数退避公式
delay = base_delay * (backoff_multiplier ^ attempt)

# 示例：base_delay=1s, multiplier=2
# 第1次重试：1s
# 第2次重试：2s
# 第3次重试：4s
# 第4次重试：8s

2.2 随机抖动（Jitter）

在指数退避的基础上添加随机抖动，避免多个客户端同时重试导致的"惊群效应"：

import random

def calculate_delay(attempt, base_delay, max_delay, jitter=True):
    """计算重试延迟时间"""
    delay = base_delay * (2 ** attempt)
    delay = min(delay, max_delay)
    
    if jitter:
        # 添加0.5x ~ 1.5x的随机抖动
        jitter_factor = random.uniform(0.5, 1.5)
        delay *= jitter_factor
    
    return delay
⚠️ 注意：在高并发场景下，如果所有客户端在同一时刻重试，会导致服务器瞬间过载。随机抖动可以有效分散重试时间。

2.3 完整实现

from dataclasses import dataclass
from typing import Callable, TypeVar, Tuple, Type
import asyncio
import random

T = TypeVar('T')

@dataclass
class RetryPolicy:
    """重试策略配置"""
    max_attempts: int = 3
    base_delay: float = 1.0
    max_delay: float = 60.0
    backoff_multiplier: float = 2.0
    jitter: bool = True
    retryable_exceptions: Tuple[Type[Exception], ...] = (
        ConnectionError,
        TimeoutError,
    )
    
    def calculate_delay(self, attempt: int) -> float:
        """计算第attempt次重试的延迟时间"""
        delay = self.base_delay * (self.backoff_multiplier ** attempt)
        delay = min(delay, self.max_delay)
        
        if self.jitter:
            jitter_factor = random.uniform(0.5, 1.5)
            delay *= jitter_factor
        
        return delay
    
    def should_retry(self, error: Exception, attempt: int) -> bool:
        """判断是否应该重试"""
        if attempt >= self.max_attempts:
            return False
        
        return isinstance(error, self.retryable_exceptions)


def retry_with_policy(policy: RetryPolicy):
    """重试装饰器"""
    def decorator(func: Callable[..., T]) -> Callable[..., T]:
        async def wrapper(*args, **kwargs) -> T:
            last_error = None
            
            for attempt in range(policy.max_attempts):
                try:
                    return await func(*args, **kwargs)
                except Exception as e:
                    last_error = e
                    
                    if not policy.should_retry(e, attempt):
                        raise
                    
                    delay = policy.calculate_delay(attempt)
                    print(f"尝试 {attempt + 1} 失败: {e}，{delay:.2f}s后重试")
                    await asyncio.sleep(delay)
            
            raise RetryableError(last_error, policy.max_attempts)
        
        return wrapper
    return decorator

三、限流器（Rate Limiter）

3.1 令牌桶算法

令牌桶是最常用的限流算法，工作原理：

  • 桶中有固定数量的令牌
  • 每个请求消耗一个令牌
  • 令牌以固定速率补充
  • 桶满时不再补充
import time
import asyncio

class TokenBucket:
    """令牌桶限流器"""
    
    def __init__(self, capacity: int, refill_rate: float):
        """
        Args:
            capacity: 桶容量（最大令牌数）
            refill_rate: 补充速率（令牌/秒）
        """
        self.capacity = capacity
        self.refill_rate = refill_rate
        self.tokens = capacity
        self.last_refill = time.time()
    
    def acquire(self, tokens: int = 1) -> bool:
        """尝试获取令牌"""
        # 补充令牌
        now = time.time()
        elapsed = now - self.last_refill
        new_tokens = elapsed * self.refill_rate
        self.tokens = min(self.capacity, self.tokens + new_tokens)
        self.last_refill = now
        
        # 检查是否有足够令牌
        if self.tokens >= tokens:
            self.tokens -= tokens
            return True
        
        return False
    
    async def acquire_or_wait(self, tokens: int = 1, timeout: float = None):
        """等待直到获取令牌或超时"""
        start_time = time.time()
        
        while True:
            if self.acquire(tokens):
                return True
            
            if timeout and (time.time() - start_time) >= timeout:
                raise RateLimitExceeded("获取令牌超时")
            
            await asyncio.sleep(0.1)

3.2 滑动窗口算法

滑动窗口算法精确控制时间窗口内的请求数：

from collections import deque

class SlidingWindow:
    """滑动窗口限流器"""
    
    def __init__(self, max_requests: int, window_size: float):
        self.max_requests = max_requests
        self.window_size = window_size
        self.requests = deque()
    
    def acquire(self) -> bool:
        """尝试记录请求"""
        now = time.time()
        window_start = now - self.window_size
        
        # 移除过期记录
        while self.requests and self.requests[0] < window_start:
            self.requests.popleft()
        
        # 检查是否超限
        if len(self.requests) < self.max_requests:
            self.requests.append(now)
            return True
        
        return False
💡 选择建议：
  • 令牌桶：适合控制突发流量，允许一定程度的流量突发
  • 滑动窗口：适合精确控制持续流量，严格按照时间窗口限制
  • 组合使用：生产环境通常组合使用，令牌桶控制突发，滑动窗口控制持续

四、熔断器（Circuit Breaker）

4.1 三种状态

状态 行为 转换条件
关闭（CLOSED） 正常处理请求，记录失败次数 失败次数达到阈值 → 打开
打开（OPEN） 快速失败，拒绝所有请求 经过恢复时间 → 半开
半开（HALF_OPEN） 允许少量请求测试服务 成功 → 关闭；失败 → 打开

4.2 熔断器核心实现

from enum import Enum
import time

class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    """熔断器"""
    
    def __init__(
        self,
        failure_threshold: int = 5,
        recovery_timeout: float = 30.0,
        success_threshold: int = 2,
    ):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.success_threshold = success_threshold
        
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.success_count = 0
        self.open_time = None
    
    async def call(self, func, *args, **kwargs):
        """通过熔断器调用函数"""
        # 检查状态
        if self.state == CircuitState.OPEN:
            if self._should_attempt_reset():
                self._transition_to_half_open()
            else:
                raise CircuitBreakerOpenError("熔断器已打开")
        
        # 执行函数
        try:
            result = await func(*args, **kwargs)
            await self._on_success()
            return result
        except Exception as e:
            await self._on_failure()
            raise
    
    def _should_attempt_reset(self) -> bool:
        """判断是否应该尝试重置"""
        if self.open_time is None:
            return False
        elapsed = time.time() - self.open_time
        return elapsed >= self.recovery_timeout
    
    def _transition_to_half_open(self):
        self.state = CircuitState.HALF_OPEN
        self.success_count = 0
    
    def _transition_to_open(self):
        self.state = CircuitState.OPEN
        self.open_time = time.time()
    
    def _transition_to_closed(self):
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.open_time = None
    
    async def _on_success(self):
        if self.state == CircuitState.HALF_OPEN:
            self.success_count += 1
            if self.success_count >= self.success_threshold:
                self._transition_to_closed()
        elif self.state == CircuitState.CLOSED:
            self.failure_count = 0
    
    async def _on_failure(self):
        self.failure_count += 1
        
        if self.state == CircuitState.HALF_OPEN:
            self._transition_to_open()
        elif self.state == CircuitState.CLOSED:
            if self.failure_count >= self.failure_threshold:
                self._transition_to_open()

五、最佳实践

5.1 组合使用三种模式

生产环境中，通常组合使用重试、限流和熔断器：

class ResilientAPIClient:
    """韧性API客户端"""
    
    def __init__(self):
        # 重试策略：最多3次，指数退避
        self.retry_policy = RetryPolicy(
            max_attempts=3,
            base_delay=1.0,
            backoff_multiplier=2.0,
        )
        
        # 令牌桶限流：容量10，每秒补充5个
        self.rate_limiter = TokenBucket(capacity=10, refill_rate=5.0)
        
        # 熔断器：5次失败后打开，30秒后半开
        self.circuit_breaker = CircuitBreaker(
            failure_threshold=5,
            recovery_timeout=30.0,
            success_threshold=2,
        )
    
    async def call_api(self, func, *args, **kwargs):
        """带完整保护的API调用"""
        # 1. 检查熔断器
        if self.circuit_breaker.state == CircuitState.OPEN:
            raise CircuitBreakerOpenError("服务暂时不可用")
        
        # 2. 检查限流
        if not self.rate_limiter.acquire():
            raise RateLimitExceeded("请求过于频繁")
        
        # 3. 通过熔断器执行（带重试）
        @retry_with_policy(self.retry_policy)
        async def execute():
            return await func(*args, **kwargs)
        
        return await self.circuit_breaker.call(execute)

5.2 错误分类与处理策略

错误类型 可重试 处理策略 示例
网络超时 ✅ 是 指数退避重试 ConnectionError, TimeoutError
服务器错误 ✅ 是 重试 + 熔断器 HTTP 500, 502, 503
请求限流 ✅ 是 等待 + 限流器 HTTP 429
认证失败 ❌ 否 更新凭证后重试 HTTP 401, 403
参数错误 ❌ 否 修正参数后重试 HTTP 400, ValueError
资源不存在 ❌ 否 不重试，返回错误 HTTP 404

5.3 配置建议

⚠️ 生产环境配置建议：
  • 重试次数：3-5次，过多会浪费资源
  • 基础延迟：1-2秒，根据API响应时间调整
  • 最大延迟：30-60秒，避免等待过长
  • 退避倍数：2.0，平衡重试频率和资源消耗
  • 熔断阈值：5-10次失败，根据服务稳定性调整
  • 恢复时间：30-60秒，根据服务恢复能力调整
  • 限流参数：根据API配额设置，预留20%缓冲
