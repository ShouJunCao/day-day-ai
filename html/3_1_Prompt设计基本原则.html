3.1 Prompt 设计基本原则

模块三：Prompt Engineering 预计学习：45 分钟

学习目标

  • 掌握 Prompt 设计的四大核心原则：清晰、具体、结构化、分步骤
  • 学会使用结构化模板组织 Prompt 内容
  • 理解好 Prompt 与坏 Prompt 的本质差异
  • 能够构建可复用的 Prompt 工程化模板系统

一、Prompt 工程概述

Prompt Engineering（提示工程）是大模型应用开发中最核心的技能之一。与大模型交互时，Prompt 的质量直接决定了输出的质量和可靠性。一个好的 Prompt 就像一份精确的"任务说明书"，它告诉模型：你是谁、要做什么、怎么做、输出什么格式。

Prompt 设计遵循四大基本原则，这四项原则构成了所有高级 Prompt 技术（如 CoT、Few-shot、ReAct 等）的基础。掌握这些原则后，后续学习高级技巧时会事半功倍。

二、清晰原则（Clear）

清晰是 Prompt 设计的第一要务。模型无法读取你的心思，它只能根据你写在 Prompt 中的文字来理解任务。模糊的表述会导致模型"猜测"你的意图，输出结果自然不可靠。

❌ 模糊的 Prompt

帮我写点东西

分析一下这个

优化这段代码

✅ 清晰的 Prompt

为我的 SaaS 产品写一篇 500 字的着陆页文案

分析这份销售数据，找出 Q3 同比下降的三个原因

将这段 O(n²) 排序改为 O(n log n)，保持接口不变

清晰的关键在于消除歧义。你应该明确告诉模型：任务类型（生成/分析/翻译/总结）、主题范围（什么领域的内容）、预期行为（模型应该做什么）。在代码中，我们通常用 role 和 task_type 来参数化这些要素。

"""
prompt_principles.py — Prompt 设计四大原则示例
清晰、具体、结构化、分步骤的实际应用
"""
from dataclasses import dataclass, field
from typing import Optional


@dataclass
class PromptBuilder:
    """结构化 Prompt 构建器，遵循四大原则"""

    role: str = ""
    task: str = ""
    context: str = ""
    constraints: list = field(default_factory=list)
    output_format: str = ""
    steps: list = field(default_factory=list)
    examples: list = field(default_factory=list)

    def build(self) -> str:
        """将各模块组装为完整 Prompt"""
        parts = []

        # 角色设定（清晰原则）
        if self.role:
            parts.append(f"## 角色\n你是一个{self.role}。")

        # 任务描述（清晰 + 具体原则）
        if self.task:
            parts.append(f"## 任务\n{self.task}")

        # 上下文（具体原则）
        if self.context:
            parts.append(f"## 背景信息\n{self.context}")

        # 约束条件（具体原则）
        if self.constraints:
            items = "\n".join(f"- {c}" for c in self.constraints)
            parts.append(f"## 约束条件\n{items}")

        # 分步骤指引（分步骤原则）
        if self.steps:
            numbered = "\n".join(
                f"{i+1}. {s}" for i, s in enumerate(self.steps)
            )
            parts.append(f"## 执行步骤\n{numbered}")

        # 输出格式（结构化原则）
        if self.output_format:
            parts.append(f"## 输出格式\n{self.output_format}")

        # 示例（Few-shot）
        if self.examples:
            example_text = "\n\n".join(
                f"示例 {i+1}:\n{e}" for i, e in enumerate(self.examples)
            )
            parts.append(f"## 示例\n{example_text}")

        return "\n\n".join(parts)

三、具体原则（Specific）

具体原则要求你为模型提供足够的约束条件和细节信息。没有约束的 Prompt 就像没有需求文档的开发任务——模型会自由发挥，结果往往不符合预期。

具体化包括以下几个维度：长度约束（多少字、多少条）、格式约束（JSON / Markdown / 表格）、风格约束（正式/口语/技术）、范围约束（只涉及哪些内容）、禁止项（不能做什么）。

    # 具体原则示例：通过约束条件精确定义期望输出
    builder = PromptBuilder(
        role="资深数据分析师",
        task="分析以下电商销售数据，生成月度分析报告",
        context="数据来源：公司 CRM 系统，时间范围：2024年1月",
        constraints=[
            "报告长度控制在 800-1200 字",
            "必须包含同比和环比数据对比",
            "使用 Markdown 表格展示关键指标",
            "不要使用专业术语，面向管理层汇报",
            "至少给出 3 条可执行的改进建议",
        ],
        output_format="Markdown 格式，包含：摘要、数据概览、问题分析、建议"
    )
    prompt = builder.build()
    print(prompt)
💡 实用技巧

约束条件越多越具体，模型的输出越可控。但也不要过度约束——给模型留一些发挥空间，让它能利用自身的知识来补充你可能遗漏的视角。

四、结构化原则（Structured）

结构化原则要求使用明确的格式组织 Prompt 内容，而不是写一大段自由文本。结构化的 Prompt 更容易被模型理解，也更容易被人类维护和复用。

常用的结构化方式包括：使用 Markdown 标题（##）划分段落、使用 XML 标签（<context>、<instruction>）包裹不同类型内容、使用 JSON/YAML 定义配置项。在工程化实践中，我们通常将 Prompt 模板存储在文件中，通过模板引擎动态填充变量。

from string import Template
from pathlib import Path
from typing import Any


class PromptTemplate:
    """基于文件的可复用 Prompt 模板系统"""

    def __init__(self, template_str: str):
        """
        初始化模板。
        template_str: 包含 $variable 占位符的模板字符串
        """
        self._template = Template(template_str)
        self._variables = self._extract_variables(template_str)

    @staticmethod
    def _extract_variables(template_str: str) -> list[str]:
        """提取模板中所有变量名"""
        import re
        return re.findall(r'\$(\w+)', template_str)

    @classmethod
    def from_file(cls, path: str | Path) -> "PromptTemplate":
        """从文件加载模板"""
        content = Path(path).read_text(encoding="utf-8")
        return cls(content)

    def render(self, **kwargs: Any) -> str:
        """渲染模板，填充变量"""
        # 检查是否所有变量都已提供
        missing = set(self._variables) - set(kwargs.keys())
        if missing:
            raise ValueError(f"缺少模板变量: {missing}")
        return self._template.safe_substitute(**kwargs)

    @property
    def variables(self) -> list[str]:
        """返回模板所需的所有变量名"""
        return list(self._variables)


# 使用示例：定义一个代码审查模板
CODE_REVIEW_TEMPLATE = """## 角色
你是一个有 10 年经验的 $language 高级工程师，专注于代码审查。

## 任务
请审查以下代码，给出专业的审查意见。

## 待审查代码
```$language
$code
```

## 审查维度
1. **代码质量**：可读性、命名规范、代码风格
2. **潜在 Bug**：边界条件、空值处理、异常安全
3. **性能**：算法复杂度、内存使用、不必要的计算
4. **安全性**：注入风险、敏感数据暴露、权限校验

## 输出要求
- 按严重程度分级：🔴 严重 / 🟡 警告 / 🟢 建议
- 每条意见包含：问题位置、问题描述、修复建议
- 最后给出总体评分（1-10 分）和总结
"""

reviewer = PromptTemplate(CODE_REVIEW_TEMPLATE)
result = reviewer.render(
    language="python",
    code="def calc(x,y): return x/y"
)
print(result[:200])  # 预览前 200 字符

五、分步骤原则（Step-by-step）

当任务比较复杂时，将其拆解为明确的执行步骤可以显著提高输出质量。这不仅帮助模型按顺序思考，也让输出的结构更可预测。这正是思维链（Chain-of-Thought）技术的基础——后面 3.3 节会深入讲解。

分步骤的核心思想是：不要试图在一个 Prompt 中完成所有事情，而是将复杂任务分解为原子步骤，每一步都有明确的输入和输出。

    # 分步骤原则：将复杂任务拆解为有序步骤
    analysis_builder = PromptBuilder(
        role="商业分析师",
        task="分析竞品产品并输出竞品分析报告",
        steps=[
            "第一步：列出竞品的 5 个核心功能特性",
            "第二步：分析每个功能的优势和劣势",
            "第三步：与我们的产品进行逐项对比",
            "第四步：识别竞品的差异化策略",
            "第五步：提出 3 条应对建议，按优先级排序",
        ],
        constraints=[
            "每个步骤完成后输出该步骤的结论",
            "最终报告不超过 1500 字",
            "使用 SWOT 分析框架组织结论",
        ],
        output_format="Markdown 报告，每个步骤一个小节"
    )
    print(analysis_builder.build())

六、好 Prompt 与坏 Prompt 对比

下面通过表格总结四大原则在实际应用中的差异，帮助你快速建立直觉：

原则❌ 反面案例✅ 正面案例关键差异
清晰 "帮我翻译一下" "将以下中文翻译为英文，保持技术文档风格" 明确了任务类型和输出要求
具体 "写一个函数" "写一个 Python 函数，输入为列表，返回去重后的排序结果，时间复杂度 O(n log n)" 给出了输入输出和性能约束
结构化 一大段自由文本描述 使用 ## 标题 + 列表 + 代码块组织 模型更容易解析结构化内容
分步骤 "分析这个市场" "1.分析市场规模 2.识别竞争对手 3.评估趋势 4.给出建议" 拆解为可执行的原子步骤

七、工程化 Prompt 管理

在实际项目中，Prompt 不是写死在代码里的。我们需要一套Prompt 管理系统来支持模板复用、版本管理和 A/B 测试。下面是一个轻量级的 Prompt 管理方案：

import json
from pathlib import Path
from datetime import datetime
from typing import Optional


class PromptRegistry:
    """
    Prompt 注册中心：管理多个 Prompt 模板的版本和元数据。
    支持模板注册、检索、版本历史追踪。
    """

    def __init__(self, storage_dir: str = "./prompts"):
        self.storage_dir = Path(storage_dir)
        self.storage_dir.mkdir(parents=True, exist_ok=True)
        self._registry: dict = {}
        self._load_registry()

    def _registry_path(self) -> Path:
        return self.storage_dir / "registry.json"

    def _load_registry(self) -> None:
        """从磁盘加载注册表"""
        path = self._registry_path()
        if path.exists():
            data = json.loads(path.read_text(encoding="utf-8"))
            self._registry = data.get("templates", {})

    def _save_registry(self) -> None:
        """持久化注册表到磁盘"""
        data = {
            "updated_at": datetime.now().isoformat(),
            "templates": self._registry,
        }
        self._registry_path().write_text(
            json.dumps(data, ensure_ascii=False, indent=2),
            encoding="utf-8",
        )

    def register(
        self,
        name: str,
        template: str,
        description: str = "",
        tags: Optional[list[str]] = None,
    ) -> dict:
        """
        注册一个新的 Prompt 模板。
        如果同名模板已存在，自动创建新版本。
        """
        # 确定版本号
        existing = self._registry.get(name, {})
        version = existing.get("version", 0) + 1

        # 保存模板文件
        template_file = self.storage_dir / f"{name}_v{version}.txt"
        template_file.write_text(template, encoding="utf-8")

        # 更新注册表
        self._registry[name] = {
            "name": name,
            "version": version,
            "description": description,
            "tags": tags or [],
            "template_file": str(template_file),
            "created_at": datetime.now().isoformat(),
        }
        self._save_registry()
        return self._registry[name]

    def get(self, name: str) -> Optional[PromptTemplate]:
        """获取指定名称的最新版本模板"""
        entry = self._registry.get(name)
        if not entry:
            return None
        content = Path(entry["template_file"]).read_text(
            encoding="utf-8"
        )
        return PromptTemplate(content)

    def list_templates(self) -> list[dict]:
        """列出所有已注册的模板"""
        return list(self._registry.values())
📌 小结

Prompt 设计是一项可以通过练习快速提升的技能。记住四大原则——清晰、具体、结构化、分步骤——它们是构建高质量 Prompt 的基石。在后续章节中，我们将在此基础上学习更高级的技术：Few-shot 提示、思维链（CoT）、角色设定（System Prompt）等，它们本质上都是这四大原则的延伸和组合。

八、Prompt 设计常见误区

在实践中，初学者容易犯以下几个典型错误。了解这些误区可以帮助你避免走弯路：

误区一：过度依赖模型的"理解力"

很多人认为大模型很聪明，应该能"猜出"我的意思。但事实上，模型是基于概率生成文本的，模糊的输入只会导致不可预测的输出。永远不要假设模型能读懂你的潜台词——写出来的才是真的。

误区二：一个 Prompt 想做所有事

把多个不相关的任务塞进一个 Prompt 中，会导致模型"注意力分散"，每个任务都做不好。正确的做法是遵循单一职责原则：一个 Prompt 只完成一个明确的任务。如果需要完成多个任务，应该拆分为多次调用或使用分步骤原则。

误区三：忽视输出格式约束

不指定输出格式，模型的返回内容就会每次都不一样，给下游解析带来巨大麻烦。在工程化场景中，必须明确要求输出格式——JSON、Markdown、XML 或特定的文本结构——这样后续代码才能可靠地处理模型输出。

误区四：Prompt 写死后不迭代

Prompt 设计是一个迭代优化的过程。第一版 Prompt 往往不是最优的，你需要根据实际输出不断调整措辞、增删约束、优化结构。建议建立 Prompt 版本管理机制（如前文的 PromptRegistry），记录每次修改的原因和效果。

⚠️ 注意

不同模型对 Prompt 的敏感度不同。GPT-4 级别的模型容错率较高，即使 Prompt 不够完美也能给出不错的结果；而小型模型（如 7B 参数级别）对 Prompt 的质量要求更高，必须严格遵守四大原则才能获得可靠输出。

九、实战建议：Prompt 设计清单

在编写每个 Prompt 之前，使用以下清单进行自查，确保覆盖四大原则的关键要素：

检查项原则具体问题是否满足
任务类型明确清晰模型需要做什么？（生成/分析/翻译/总结）□
角色已设定清晰模型应该扮演什么角色？□
长度约束具体输出应该多长？多少条？□
格式约束具体输出应该是什么格式？□
风格约束具体语言风格是正式还是口语化？□
禁止项具体模型不应该做什么？□
段落分隔结构化是否使用了标题或标签分隔？□
步骤拆解分步骤复杂任务是否拆为有序步骤？□
示例提供Few-shot是否给出了输入输出示例？□

养成使用清单的习惯后，你的 Prompt 质量会稳步提升。随着经验积累，很多检查项会变成下意识的行为，但在初期严格按照清单执行是非常有必要的。
