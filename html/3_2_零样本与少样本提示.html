3.2 零样本与少样本提示

模块三：Prompt Engineering 预计学习：50 分钟

学习目标

  • 理解 Zero-shot（零样本）和 Few-shot（少样本）提示的核心概念与区别
  • 掌握何时使用 Zero-shot、何时使用 Few-shot 的决策框架
  • 学会构建高质量的 Few-shot 示例集
  • 能够工程化实现动态示例选择与管理系统

一、概念解析

在大模型应用中，样本（shot）指的是提供给模型的输入-输出示例对。根据提供示例的数量，Prompt 技术分为两大类：

  • Zero-shot（零样本）：不提供任何示例，直接让模型完成任务。依赖模型预训练时学到的通用知识。
  • Few-shot（少样本）：提供 1-N 个示例（通常 3-5 个），让模型学习示例中的模式后完成任务。

二、决策框架：何时用哪种？

选择 Zero-shot 还是 Few-shot，取决于任务的复杂度、标准化程度和输出质量要求。以下决策框架可以帮助你快速判断：

场景特征推荐方式原因
任务简单、通用（翻译、摘要、问答） Zero-shot 模型预训练已充分掌握，无需额外示例
输出格式固定、有明确规范 Few-shot 示例能精确展示期望的格式和风格
领域专业性强（医疗、法律、金融） Few-shot 示例提供领域知识，减少幻觉
需要高度一致性和可预测性 Few-shot 示例约束输出范围，提高稳定性
快速原型验证、探索性任务 Zero-shot 无需准备示例，快速迭代
Token 预算有限 Zero-shot Few-shot 示例会消耗额外 Token
💡 实用建议

在实践中，可以先用 Zero-shot 快速验证想法，如果输出质量不达标或一致性不够，再切换到 Few-shot。Few-shot 的示例数量通常 3-5 个即可，过多示例不仅浪费 Token，还可能引入噪声。

三、Zero-shot 实践

Zero-shot 的关键在于清晰的任务描述和具体的约束条件。即使没有示例，通过 3.1 节的四大原则（清晰、具体、结构化、分步骤），仍然可以获得高质量的输出。

Zero-shot 最适合以下场景：通用文本处理（翻译、摘要、改写）、简单分类任务（情感分析、主题分类）、创意生成（写故事、生成创意）、快速原型验证（验证想法可行性）。在这些场景中，模型的预训练知识已经足够，额外示例反而可能限制模型的创造力。

使用 Zero-shot 时，要特别注意任务描述的精确性。模糊的描述会导致模型输出不可预测。例如，"分析这段文本"不如"分析这段文本的情感倾向，输出正面/负面/中性三分类结果"来得有效。

"""
zero_few_shot.py — Zero-shot 与 Few-shot 提示实现
集成重试、限流、日志等生产级特性
"""

from typing import Optional
from dataclasses import dataclass
import json

from llm_sdk import UnifiedLLMClient  # 复用 2.8 节的 SDK


@dataclass
class ZeroShotPrompt:
    """Zero-shot 提示构建器"""
    
    task: str
    constraints: list[str]
    output_format: str = "text"
    
    def build(self) -> str:
        """构建 Zero-shot Prompt"""
        parts = [f"## 任务\n{self.task}"]
        
        if self.constraints:
            items = "\n".join(f"- {c}" for c in self.constraints)
            parts.append(f"## 约束条件\n{items}")
        
        if self.output_format != "text":
            parts.append(f"## 输出格式\n{self.output_format}")
        
        return "\n\n".join(parts)


# Zero-shot 示例：情感分析
def zero_shot_sentiment(text: str, client: UnifiedLLMClient) -> dict:
    """
    Zero-shot 情感分析：不提供示例，直接让模型判断
    
    参数:
        text: 待分析文本
        client: LLM 客户端
    返回:
        包含情感和置信度的字典
    """
    prompt = ZeroShotPrompt(
        task=f"分析以下文本的情感倾向：\n\n{text}",
        constraints=[
            "情感类别：正面 / 负面 / 中性",
            "输出 JSON 格式",
            "包含 confidence 字段（0-1 之间）",
        ],
        output_format='{"sentiment": "正面/负面/中性", "confidence": 0.95}'
    ).build()
    
    response = client.chat(
        messages=[{"role": "user", "content": prompt}],
        temperature=0.3,  # 低温度提高一致性
    )
    
    # 解析 JSON 输出
    try:
        result = json.loads(response.content)
        return result
    except json.JSONDecodeError:
        # 如果模型输出不是标准 JSON，尝试提取
        return {"sentiment": "unknown", "raw": response.content}


# 测试 Zero-shot
client = UnifiedLLMClient(provider="openai", model="gpt-4")
result = zero_shot_sentiment("这个产品太棒了！", client)
print(result)  # {"sentiment": "正面", "confidence": 0.98}

四、Few-shot 实践

Few-shot 的核心是高质量的示例集。示例应该覆盖任务的典型场景，展示期望的输出格式和风格。构建 Few-shot Prompt 时，需要注意以下几点：

  • 代表性：示例应覆盖任务的主要类别和边界情况
  • 一致性：所有示例的输入输出格式必须统一
  • 简洁性：示例不宜过长，避免消耗过多 Token
  • 多样性：示例应展示不同的输入模式，避免过拟合
@dataclass
class Example:
    """单个 Few-shot 示例"""
    input: str
    output: str
    tags: list[str] = None  # 用于分类和筛选
    
    def __post_init__(self):
        self.tags = self.tags or []


@dataclass
class FewShotPrompt:
    """Few-shot 提示构建器"""
    
    task: str
    examples: list[Example]
    actual_input: str
    constraints: list[str] = None
    output_format: str = ""
    
    def build(self) -> str:
        """构建 Few-shot Prompt"""
        parts = [f"## 任务\n{self.task}"]
        
        # 添加示例
        if self.examples:
            example_text = []
            for i, ex in enumerate(self.examples, 1):
                example_text.append(
                    f"### 示例 {i}\n"
                    f"**输入：**\n{ex.input}\n\n"
                    f"**输出：**\n{ex.output}"
                )
            parts.append("## 示例\n" + "\n\n".join(example_text))
        
        # 实际任务
        parts.append(f"## 实际任务\n**输入：**\n{self.actual_input}")
        
        # 约束条件
        if self.constraints:
            items = "\n".join(f"- {c}" for c in self.constraints)
            parts.append(f"## 约束条件\n{items}")
        
        # 输出格式
        if self.output_format:
            parts.append(f"## 输出格式\n{self.output_format}")
        
        return "\n\n".join(parts)


# Few-shot 示例：命名实体识别（NER）
def few_shot_ner(text: str, client: UnifiedLLMClient) -> list[dict]:
    """
    Few-shot 命名实体识别：提供示例让模型学习提取模式
    
    参数:
        text: 待提取文本
        client: LLM 客户端
    返回:
        实体列表，每个实体包含 name、type、span
    """
    examples = [
        Example(
            input="苹果公司今天发布了新款 iPhone。",
            output='[{"name": "苹果公司", "type": "ORG"}, {"name": "iPhone", "type": "PRODUCT"}]',
            tags=["tech", "product"],
        ),
        Example(
            input="张三在北京大学获得了博士学位。",
            output='[{"name": "张三", "type": "PERSON"}, {"name": "北京大学", "type": "ORG"}]',
            tags=["education", "person"],
        ),
        Example(
            input="腾讯的股价上涨了 5%，达到 400 港元。",
            output='[{"name": "腾讯", "type": "ORG"}, {"name": "400 港元", "type": "MONEY"}]',
            tags=["finance", "stock"],
        ),
    ]
    
    prompt = FewShotPrompt(
        task="从文本中提取命名实体（人名、组织、产品、地点、金额等）",
        examples=examples,
        actual_input=text,
        constraints=[
            "输出 JSON 数组格式",
            "每个实体包含 name 和 type 字段",
            "type 可选值：PERSON / ORG / PRODUCT / LOCATION / MONEY / DATE",
        ],
        output_format='[{"name": "实体名", "type": "类型"}]'
    ).build()
    
    response = client.chat(
        messages=[{"role": "user", "content": prompt}],
        temperature=0.2,
    )
    
    try:
        entities = json.loads(response.content)
        return entities if isinstance(entities, list) else []
    except json.JSONDecodeError:
        return []


# 测试 Few-shot
entities = few_shot_ner(
    "李四在阿里巴巴工作，月薪 3 万元。",
    client
)
print(entities)
# [{"name": "李四", "type": "PERSON"}, {"name": "阿里巴巴", "type": "ORG"}, {"name": "3 万元", "type": "MONEY"}]

五、动态示例选择

在工程化场景中，我们通常有一个示例库，需要根据当前输入动态选择最相关的示例。这可以通过以下方式实现：

  • 基于标签匹配：根据输入的标签筛选示例
  • 基于相似度：使用向量检索找到最相似的示例
  • 基于规则：根据业务规则选择示例子集
class ExampleLibrary:
    """
    示例库管理器：支持动态选择和版本管理
    
    特性:
        - 按标签筛选示例
        - 按相似度检索示例（需集成向量数据库）
        - 示例版本控制
        - 使用统计追踪
    """
    
    def __init__(self):
        self._examples: dict[str, Example] = {}
        self._usage_stats: dict[str, int] = {}
    
    def add(self, example_id: str, example: Example) -> None:
        """添加示例到库中"""
        self._examples[example_id] = example
        self._usage_stats[example_id] = 0
    
    def remove(self, example_id: str) -> bool:
        """删除示例"""
        if example_id in self._examples:
            del self._examples[example_id]
            del self._usage_stats[example_id]
            return True
        return False
    
    def get_by_tags(self, tags: list[str], limit: int = 5) -> list[Example]:
        """
        按标签筛选示例
        
        参数:
            tags: 标签列表（示例需包含至少一个标签）
            limit: 返回数量上限
        返回:
            匹配的示例列表
        """
        matched = []
        for ex_id, example in self._examples.items():
            if any(tag in example.tags for tag in tags):
                matched.append((ex_id, example))
        
        # 按使用频率排序（优先使用低频示例，保持多样性）
        matched.sort(key=lambda x: self._usage_stats[x[0]])
        
        # 更新使用统计
        result = []
        for ex_id, example in matched[:limit]:
            self._usage_stats[ex_id] += 1
            result.append(example)
        
        return result
    
    def get_random(self, count: int = 3) -> list[Example]:
        """随机选择示例（用于基线测试）"""
        import random
        examples = list(self._examples.values())
        return random.sample(examples, min(count, len(examples)))
    
    def stats(self) -> dict:
        """返回示例库统计信息"""
        return {
            "total": len(self._examples),
            "usage": self._usage_stats,
            "tags": self._collect_tags(),
        }
    
    def _collect_tags(self) -> dict[str, int]:
        """统计所有标签的使用情况"""
        tag_counts = {}
        for example in self._examples.values():
            for tag in example.tags:
                tag_counts[tag] = tag_counts.get(tag, 0) + 1
        return tag_counts


# 使用示例库
library = ExampleLibrary()

# 添加示例
library.add("ex1", Example(
    input="这个手机拍照效果很好",
    output='{"sentiment": "正面", "aspect": "拍照"}',
    tags=["product", "positive", "camera"]
))
library.add("ex2", Example(
    input="电池续航太差了",
    output='{"sentiment": "负面", "aspect": "电池"}',
    tags=["product", "negative", "battery"]
))
library.add("ex3", Example(
    input="价格合理，性价比高",
    output='{"sentiment": "正面", "aspect": "价格"}',
    tags=["product", "positive", "price"]
))

# 动态选择示例
input_text = "屏幕显示效果很清晰"
input_tags = ["product", "screen"]  # 实际应用中可用分类器自动打标签
selected = library.get_by_tags(input_tags, limit=2)
print(f"选择了 {len(selected)} 个示例")

# 查看统计
print(library.stats())

六、Zero-shot vs Few-shot 对比

通过以下对比，你可以更清楚地理解两种方式的差异：

维度Zero-shotFew-shot
Token 消耗低较高（示例占用 Token）
输出一致性较低较高
准备成本无需准备示例需要构建示例集
适用场景通用任务、快速验证标准化任务、专业领域
对模型的依赖强依赖预训练知识通过示例补充知识
可解释性较低较高（示例即解释）
迭代速度快较慢（需调整示例）
维护成本无需要维护示例库
对新任务的适应性即时可用需要收集示例
上下文窗口占用少多

📌 实际案例：客服意图识别

Zero-shot 方案：直接描述任务，让模型判断用户意图。优点是快速上线，缺点是意图类别多时准确率下降。

Few-shot 方案：为每个意图类别提供 2-3 个示例。准确率高，但需要维护示例库。

混合方案：常见意图用 Few-shot（高频、高准确率），罕见意图用 Zero-shot（节省 Token）。这是工程实践中最常用的方式。

📊 性能数据参考

根据实际项目经验，在客服意图识别任务中：Zero-shot 准确率约 75-85%，Few-shot（3-5 个示例）准确率约 90-95%。但 Few-shot 的 Token 消耗是 Zero-shot 的 2-3 倍。选择时需要权衡准确率和成本。

七、最佳实践与常见陷阱

最佳实践

  • 示例质量优先于数量：3 个高质量示例胜过 10 个低质量示例
  • 覆盖边界情况：示例应包含正常情况和异常情况
  • 定期更新示例库：随着业务变化，示例也需要迭代
  • A/B 测试：对比 Zero-shot 和 Few-shot 的效果，用数据说话

常见陷阱

  • 示例过拟合：示例太少或太相似，导致模型只学会复制示例模式
  • 示例噪声：示例本身有错误，会误导模型输出
  • 示例过长：单个示例占用太多 Token，挤压实际任务的空间
  • 忽视 Zero-shot：不是所有任务都需要 Few-shot，简单任务用 Zero-shot 更高效
⚠️ 注意

Few-shot 示例的顺序很重要。研究表明，将最相关的示例放在最后（靠近实际任务）效果更好。如果你使用动态示例选择，建议按相似度升序排列。

八、One-shot 与 Many-shot

除了 Zero-shot 和 Few-shot，还有两种变体值得了解：

  • One-shot（单样本）：只提供一个示例。适用于任务非常简单、示例空间有限的场景。优点是节省 Token，缺点是示例的代表性可能不足。
  • Many-shot（多样本）：提供 10 个以上的示例。适用于复杂任务、需要覆盖大量边界情况的场景。但要注意 Token 消耗和上下文窗口限制。
# One-shot 示例：简单的情感分类
one_shot_examples = [
    Example(
        input="这个产品太棒了！",
        output='{"sentiment": "positive"}',
    )
]

# Many-shot 示例：复杂的多标签分类
many_shot_examples = [
    Example(
        input="电池续航差，但拍照效果好",
        output='{"negative": ["battery"], "positive": ["camera"]}',
        tags=["mixed", "product"],
    ),
    Example(
        input="价格合理，屏幕清晰，但系统卡顿",
        output='{"positive": ["price", "screen"], "negative": ["performance"]}',
        tags=["mixed", "product"],
    ),
    # ... 更多示例
]

def adaptive_shot_selection(task_complexity: str) -> int:
    """
    根据任务复杂度自适应选择示例数量
    
    参数:
        task_complexity: 任务复杂度（simple / medium / complex）
    返回:
        推荐的示例数量
    """
    complexity_map = {
        "simple": 1,      # One-shot
        "medium": 3,      # Few-shot
        "complex": 8,     # Many-shot
    }
    return complexity_map.get(task_complexity, 3)

# 使用示例
recommended_count = adaptive_shot_selection("medium")
print(f"推荐使用 {recommended_count} 个示例")

九、工程化建议

在生产环境中，Zero-shot 和 Few-shot 的选择应该基于数据驱动的决策。建议建立以下机制：

  • 评估数据集：为每种任务准备标准测试集，定期评估不同方案的效果
  • 成本监控：追踪 Zero-shot 和 Few-shot 的 Token 消耗和 API 成本
  • 质量指标：定义明确的输出质量标准（准确率、一致性、延迟等）
  • 自动切换：根据任务类型或输入特征，自动选择 Zero-shot 或 Few-shot
class PromptStrategySelector:
    """
    Prompt 策略选择器：根据任务特征自动选择 Zero-shot 或 Few-shot
    
    决策因素:
        - 任务复杂度
        - 输出一致性要求
        - Token 预算
        - 历史成功率
    """
    
    def __init__(self):
        self._success_stats = {
            "zero_shot": {"total": 0, "success": 0},
            "few_shot": {"total": 0, "success": 0},
        }
    
    def select_strategy(
        self,
        task_type: str,
        consistency_required: bool = False,
        token_budget: int = 1000,
    ) -> str:
        """
        选择 Prompt 策略
        
        参数:
            task_type: 任务类型（simple / standard / specialized）
            consistency_required: 是否需要高一致性
            token_budget: Token 预算
        返回:
            策略名称（"zero_shot" 或 "few_shot"）
        """
        # 规则 1：高一致性要求 → Few-shot
        if consistency_required:
            return "few_shot"
        
        # 规则 2：专业领域任务 → Few-shot
        if task_type == "specialized":
            return "few_shot"
        
        # 规则 3：Token 预算紧张 → Zero-shot
        if token_budget < 500:
            return "zero_shot"
        
        # 规则 4：简单任务 → Zero-shot
        if task_type == "simple":
            return "zero_shot"
        
        # 默认：标准任务用 Few-shot
        return "few_shot"
    
    def record_result(self, strategy: str, success: bool) -> None:
        """记录策略执行结果"""
        self._success_stats[strategy]["total"] += 1
        if success:
            self._success_stats[strategy]["success"] += 1
    
    def get_success_rate(self, strategy: str) -> float:
        """获取策略的历史成功率"""
        stats = self._success_stats[strategy]
        if stats["total"] == 0:
            return 0.0
        return stats["success"] / stats["total"]


# 使用示例
selector = PromptStrategySelector()

# 场景 1：简单翻译任务
strategy1 = selector.select_strategy(
    task_type="simple",
    consistency_required=False,
    token_budget=800,
)
print(f"翻译任务策略: {strategy1}")  # zero_shot

# 场景 2：医疗报告生成
strategy2 = selector.select_strategy(
    task_type="specialized",
    consistency_required=True,
    token_budget=2000,
)
print(f"医疗报告策略: {strategy2}")  # few_shot

# 记录结果并查看成功率
selector.record_result("zero_shot", success=True)
selector.record_result("zero_shot", success=True)
selector.record_result("few_shot", success=True)
print(f"Zero-shot 成功率: {selector.get_success_rate('zero_shot'):.1%}")
