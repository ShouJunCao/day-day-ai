3.3 思维链（CoT）与思维树（Tree-of-Thought）

模块三：Prompt Engineering 预计学习：55 分钟

学习目标

  • 理解 Chain-of-Thought（CoT） 的核心原理与变体
  • 掌握 Tree-of-Thought（ToT） 的搜索框架与实现
  • 学会在工程化场景中选择合适的推理策略
  • 能够构建生产级的 CoT / ToT 推理引擎

一、为什么需要推理引导？

大模型在简单任务上表现出色，但面对复杂推理任务（数学计算、逻辑推导、多步骤分析）时，直接给出答案往往不够准确。原因在于：模型是基于概率生成下一个 token 的，复杂推理需要多步逻辑链条，而一次性生成容易在中间步骤出错。

研究表明，如果引导模型逐步展示推理过程，而非直接跳到最终答案，输出质量会显著提升。这就是思维链（Chain-of-Thought）的核心思想——让模型"先想清楚再给出回答"。

❌ 直接回答

问题：一个商店有 23 个苹果，卖了 17 个，又进了 6 个，现在有多少？

回答：12 个

❌ 错误（正确答案是 12... 等等，23-17+6=12，这次碰巧对了，但更复杂的问题就容易出错）

✅ CoT 逐步推理

思考过程：

1. 初始：23 个苹果

2. 卖了 17 个：23 - 17 = 6 个

3. 又进了 6 个：6 + 6 = 12 个

答案：12 个 ✅

二、Chain-of-Thought（CoT）

CoT 由 Wei et al.（2022）提出，核心思想是在 Prompt 中加入逐步推理的示例，让模型学习展示中间步骤。CoT 有几种主要变体：

  • Few-shot CoT：在 Few-shot 示例中包含完整的推理步骤
  • Zero-shot CoT：仅通过 "Let's think step by step" 触发推理
  • Self-Ask：模型自己提出子问题并逐一回答
  • Least-to-Most：将复杂问题拆解为从简到难的子问题链

三、Zero-shot CoT 实践

Zero-shot CoT 是最简单也最实用的推理引导方式——只需在 Prompt 末尾加上"Let's think step by step"（或中文的"让我们一步步思考"），模型就会自动展示推理过程。这个技巧虽然简单，但在数学推理、逻辑分析等任务上效果显著。

"""
cot_tot.py — 思维链（CoT）与思维树（ToT）实现
包含 Zero-shot CoT、Few-shot CoT、ToT 搜索框架
"""

from typing import Optional, Callable
from dataclasses import dataclass, field
import json


@dataclass
class CoTResult:
    """CoT 推理结果"""
    question: str
    reasoning_steps: list[str]
    final_answer: str
    confidence: float = 0.0


class ZeroShotCoT:
    """
    Zero-shot CoT：通过触发词引导模型逐步推理。
    
    核心原理：
        在 Prompt 中加入 "Let's think step by step"
        触发模型的推理链，无需提供示例。
    
    适用场景：
        数学推理、逻辑分析、常识推理
    """
    
    TRIGGERS = {
        "en": "Let's think step by step.",
        "zh": "让我们一步步思考。",
        "detailed": "Please break this down step by step, "
                    "showing your reasoning at each step.",
    }
    
    def __init__(self, language: str = "zh"):
        """
        参数:
            language: 触发词语言（en / zh / detailed）
        """
        self.trigger = self.TRIGGERS.get(language, self.TRIGGERS["zh"])
    
    def build_prompt(self, question: str) -> str:
        """
        构建 Zero-shot CoT Prompt。
        
        参数:
            question: 需要推理的问题
        返回:
            包含触发词的完整 Prompt
        """
        return (
            f"## 问题\n{question}\n\n"
            f"## 要求\n"
            f"请先展示完整的推理过程（每一步都要写清楚），"
            f"最后给出最终答案。\n\n"
            f"{self.trigger}"
        )
    
    def parse_response(self, response_text: str) -> CoTResult:
        """
        解析模型输出，提取推理步骤和最终答案。
        
        参数:
            response_text: 模型的完整输出
        返回:
            CoTResult 对象
        """
        lines = response_text.strip().split("\n")
        steps = []
        answer = ""
        
        for line in lines:
            stripped = line.strip()
            # 提取编号步骤（如 "1. xxx"、"步骤1："）
            if any(stripped.startswith(p) for p in 
                   ["1.", "2.", "3.", "4.", "5.", "6.",
                    "步骤", "Step", "第"]):
                steps.append(stripped)
            # 提取最终答案
            if "答案" in stripped or "answer" in stripped.lower():
                answer = stripped
        
        return CoTResult(
            question="",
            reasoning_steps=steps,
            final_answer=answer or lines[-1] if lines else "",
        )

四、Few-shot CoT 实践

Few-shot CoT 在示例中展示完整的推理过程，比 Zero-shot CoT 更加可控。通过精心设计的示例，你可以指定推理的粒度、格式和深度。关键是在示例中不仅展示答案，还要展示思考过程。

@dataclass
class CoTExample:
    """CoT 示例：包含问题、推理步骤和答案"""
    question: str
    steps: list[str]
    answer: str


class FewShotCoT:
    """
    Few-shot CoT：通过包含推理步骤的示例引导模型。
    
    相比 Zero-shot CoT 的优势：
        - 推理格式更可控
        - 推理深度更一致
        - 适合特定领域的推理模式
    """
    
    def __init__(self, examples: list[CoTExample]):
        """
        参数:
            examples: CoT 示例列表（建议 2-4 个）
        """
        self.examples = examples
    
    def build_prompt(self, question: str) -> str:
        """
        构建 Few-shot CoT Prompt。
        
        参数:
            question: 需要推理的实际问题
        返回:
            包含示例和问题的完整 Prompt
        """
        parts = ["## 任务\n请像下面的示例一样，逐步推理并给出答案。\n"]
        
        # 添加示例
        for i, ex in enumerate(self.examples, 1):
            steps_text = "\n".join(
                f"  {j+1}. {s}" for j, s in enumerate(ex.steps)
            )
            parts.append(
                f"### 示例 {i}\n"
                f"**问题：** {ex.question}\n"
                f"**推理过程：**\n{steps_text}\n"
                f"**答案：** {ex.answer}"
            )
        
        # 实际问题
        parts.append(
            f"### 实际任务\n"
            f"**问题：** {question}\n"
            f"**推理过程：**"
        )
        
        return "\n\n".join(parts)
    
    @classmethod
    def math_reasoning(cls) -> "FewShotCoT":
        """创建数学推理的预设 CoT 模板"""
        examples = [
            CoTExample(
                question="小明有 15 个糖果，给了小红 7 个，"
                         "又从小华那里得到 3 个，现在有几个？",
                steps=[
                    "初始数量：15 个糖果",
                    "给小红 7 个：15 - 7 = 8 个",
                    "从小华得到 3 个：8 + 3 = 11 个",
                ],
                answer="11 个",
            ),
            CoTExample(
                question="一个水池，A 管 6 小时注满，B 管 8 小时注满，"
                         "两管同时开，几小时注满？",
                steps=[
                    "A 管每小时注水量：1/6",
                    "B 管每小时注水量：1/8",
                    "两管同时开，每小时注水量：1/6 + 1/8 = 7/24",
                    "注满时间：1 ÷ (7/24) = 24/7 ≈ 3.43 小时",
                ],
                answer="约 3.43 小时（24/7 小时）",
            ),
        ]
        return cls(examples)
💡 选择建议

Zero-shot CoT 适合快速验证和通用推理；Few-shot CoT 适合需要统一推理格式的生产场景。在实践中，可以先用 Zero-shot CoT 验证可行性，确认可行后再设计 Few-shot CoT 模板以提高稳定性。

五、Tree-of-Thought（ToT）

CoT 的局限在于它只探索一条推理路径。如果中间某一步出错，整个推理链条就会偏离。Tree-of-Thought（Yao et al., 2023）通过同时探索多条推理路径，并在每个步骤评估和选择最优方向，显著提高了复杂推理的准确率。

ToT 的核心流程分为四步：分解（将问题拆为子步骤）→ 生成（每步生成多个候选方案）→ 评估（用 LLM 打分）→ 搜索（BFS/DFS 找最优路径）。这个过程类似人类解决复杂问题时的"头脑风暴"——先发散思考多种可能，再收敛到最优方案。

ToT 的搜索策略通常采用 beam search（束搜索）：在每一步保留 Top-K 条最优路径，而不是只保留一条。这既保证了搜索的广度，又避免了指数级的计算开销。搜索深度（max_depth）和分支因子（branching_factor）是两个关键超参数，需要根据任务复杂度调优。

class TreeOfThought:
    """
    Tree-of-Thought 搜索框架。
    
    通过 BFS/DFS 在多个推理路径中搜索最优解。
    每个节点代表一个推理步骤，LLM 负责生成候选
    和评估分数。
    
    参数:
        llm_client: LLM 客户端
        max_depth: 最大推理深度
        branching_factor: 每步生成的候选数
        search_method: 搜索方法（bfs / dfs）
    """
    
    def __init__(
        self,
        llm_client,
        max_depth: int = 5,
        branching_factor: int = 3,
        search_method: str = "bfs",
    ):
        self.client = llm_client
        self.max_depth = max_depth
        self.branching = branching_factor
        self.search_method = search_method
    
    def generate_candidates(
        self, question: str, current_path: list[str]
    ) -> list[str]:
        """
        在当前推理路径上生成候选下一步。
        
        参数:
            question: 原始问题
            current_path: 已有的推理步骤
        返回:
            候选推理步骤列表
        """
        path_text = "\n".join(
            f"  步骤 {i+1}: {s}" for i, s in enumerate(current_path)
        ) or "  （尚未开始）"
        
        prompt = (
            f"问题：{question}\n\n"
            f"已有推理步骤：\n{path_text}\n\n"
            f"请给出 {self.branching} 种可能的下一步推理，"
            f"每种用一行描述，格式为 JSON 数组。"
        )
        
        response = self.client.chat(
            messages=[{"role": "user", "content": prompt}],
            temperature=0.8,  # 高温度增加多样性
        )
        
        try:
            candidates = json.loads(response.content)
            return candidates if isinstance(candidates, list) else []
        except json.JSONDecodeError:
            return []
    
    def evaluate(
        self, question: str, path: list[str]
    ) -> float:
        """
        评估当前推理路径的质量（0-1 分）。
        
        参数:
            question: 原始问题
            path: 推理步骤列表
        返回:
            路径质量分数（0-1）
        """
        path_text = "\n".join(
            f"  步骤 {i+1}: {s}" for i, s in enumerate(path)
        )
        
        prompt = (
            f"问题：{question}\n\n"
            f"推理过程：\n{path_text}\n\n"
            f"请评估这个推理过程的正确性和合理性，"
            f"输出一个 0-1 之间的分数（仅输出数字）。"
        )
        
        response = self.client.chat(
            messages=[{"role": "user", "content": prompt}],
            temperature=0.1,  # 低温度提高评估一致性
        )
        
        try:
            score = float(response.content.strip())
            return max(0.0, min(1.0, score))
        except (ValueError, AttributeError):
            return 0.0
    
    def solve(self, question: str) -> dict:
        """
        使用 BFS 搜索最优推理路径。
        
        参数:
            question: 需要推理的问题
        返回:
            包含最优路径和分数的字典
        """
        # BFS：维护一个优先队列
        paths = [([], 0.0)]  # (path, score)
        best_path = []
        best_score = 0.0
        
        for depth in range(self.max_depth):
            new_paths = []
            
            for path, _ in paths:
                # 生成候选
                candidates = self.generate_candidates(
                    question, path
                )
                
                for candidate in candidates:
                    new_path = path + [candidate]
                    score = self.evaluate(question, new_path)
                    new_paths.append((new_path, score))
                    
                    if score > best_score:
                        best_score = score
                        best_path = new_path
            
            # 保留 Top-K 路径（beam search）
            new_paths.sort(key=lambda x: x[1], reverse=True)
            paths = new_paths[:self.branching]
            
            # 如果最高分已经达到阈值，提前终止
            if best_score >= 0.9:
                break
        
        return {
            "question": question,
            "best_path": best_path,
            "score": best_score,
            "steps_explored": len(paths) * self.max_depth,
        }

六、CoT 与 ToT 对比

理解两种方法的差异，有助于在实际场景中做出正确选择：

维度CoT（思维链）ToT（思维树）
推理结构线性链条树状搜索
路径数量单条路径多条路径并行探索
API 调用次数1 次N × B 次（N=深度, B=分支数）
Token 成本低高（多次调用）
准确率中等高（搜索最优路径）
延迟低高
实现复杂度简单复杂
适用场景简单推理、快速响应复杂问题、高准确率要求
⚠️ 成本提醒

ToT 的 API 调用次数是 CoT 的 10-50 倍。例如 max_depth=5, branching_factor=3 时，单次 ToT 搜索需要约 15 次 API 调用。在高频场景中，务必设置调用上限和超时机制，避免成本失控。

七、工程化集成

在生产环境中，我们需要将 CoT 和 ToT 集成到统一的推理引擎中，并根据任务复杂度自动选择策略。以下是生产级的推理引擎实现：

class ReasoningEngine:
    """
    统一推理引擎：自动选择 CoT / ToT 策略。
    
    集成重试、限流、日志等生产级特性。
    支持任务复杂度自动评估和策略切换。
    
    参数:
        llm_client: LLM 客户端
        default_strategy: 默认策略（auto / cot / tot）
        max_api_calls: 单次推理最大 API 调用次数
    """
    
    def __init__(
        self,
        llm_client,
        default_strategy: str = "auto",
        max_api_calls: int = 20,
    ):
        self.client = llm_client
        self.strategy = default_strategy
        self.max_calls = max_api_calls
        self._call_count = 0
    
    def assess_complexity(self, question: str) -> str:
        """
        评估问题复杂度，返回推荐策略。
        
        评估维度：
            - 是否涉及多步计算
            - 是否有多个约束条件
            - 是否需要逻辑推导
        """
        prompt = (
            f"评估以下问题的推理复杂度，输出 'simple' 或 "
            f"'complex'（仅输出一个词）：\n\n{question}"
        )
        response = self.client.chat(
            messages=[{"role": "user", "content": prompt}],
            temperature=0.1,
        )
        self._call_count += 1
        
        result = response.content.strip().lower()
        return "complex" if "complex" in result else "simple"
    
    def reason(self, question: str) -> dict:
        """
        执行推理，自动选择最优策略。
        
        参数:
            question: 需要推理的问题
        返回:
            包含推理结果和元数据的字典
        """
        self._call_count = 0
        
        # 确定策略
        if self.strategy == "auto":
            complexity = self.assess_complexity(question)
            strategy = "tot" if complexity == "complex" else "cot"
        else:
            strategy = self.strategy
        
        # 执行推理
        if strategy == "cot":
            cot = ZeroShotCoT(language="zh")
            prompt = cot.build_prompt(question)
            response = self.client.chat(
                messages=[{"role": "user", "content": prompt}],
            )
            self._call_count += 1
            result = cot.parse_response(response.content)
            return {
                "strategy": "cot",
                "question": question,
                "answer": result.final_answer,
                "steps": result.reasoning_steps,
                "api_calls": self._call_count,
            }
        else:
            tot = TreeOfThought(
                self.client,
                max_depth=3,
                branching_factor=2,
            )
            result = tot.solve(question)
            self._call_count += result["steps_explored"]
            return {
                "strategy": "tot",
                "question": question,
                "answer": result["best_path"],
                "score": result["score"],
                "api_calls": self._call_count,
            }

八、推理策略选择指南

以下决策框架帮助你根据实际场景选择最合适的推理策略：

场景推荐策略理由
简单数学题Zero-shot CoT一步触发即可，成本低
标准化推理任务Few-shot CoT格式可控，输出一致
复杂逻辑推导ToT (BFS)多路径搜索，容错性高
创意/开放性问题ToT + 高温度探索多样性方案
实时响应场景Zero-shot CoT延迟低，一次调用
高准确率要求ToT + Self-Consistency多次验证，投票决策

九、CoT 高级技巧

掌握基础的 CoT 之后，还有一些进阶技巧可以进一步提升推理质量：

技巧一：分而治之（Decomposition）

对于复杂问题，先让模型将问题拆解为子问题，再对每个子问题分别进行 CoT 推理。这种方式比直接对整个问题做 CoT 更可靠，因为每个子问题的推理链条更短，出错概率更低。

class DecomposedCoT:
    """
    分而治之 CoT：先拆解问题，再逐步推理。
    
    工作流程：
        1. 将复杂问题拆解为 N 个子问题
        2. 对每个子问题独立执行 CoT
        3. 汇总各子问题的答案，得出最终结论
    
    优势：
        - 每个子推理链更短，出错概率低
        - 子问题可并行处理
        - 中间结果可复用和缓存
    """
    
    def __init__(self, llm_client):
        self.client = llm_client
    
    def decompose(self, question: str) -> list[str]:
        """将复杂问题拆解为子问题列表"""
        prompt = (
            f"请将以下复杂问题拆解为 2-4 个简单的子问题，"
            f"输出 JSON 数组格式：\n\n{question}"
        )
        response = self.client.chat(
            messages=[{"role": "user", "content": prompt}],
            temperature=0.2,
        )
        try:
            sub_questions = json.loads(response.content)
            return sub_questions if isinstance(sub_questions, list) else []
        except json.JSONDecodeError:
            return [question]  # 降级为原问题
    
    def solve_sub_question(self, sub_q: str, context: str = "") -> str:
        """对单个子问题执行 CoT"""
        cot = ZeroShotCoT(language="zh")
        prompt = cot.build_prompt(sub_q)
        if context:
            prompt = f"## 已知信息\n{context}\n\n{prompt}"
        
        response = self.client.chat(
            messages=[{"role": "user", "content": prompt}],
        )
        return response.content
    
    def solve(self, question: str) -> dict:
        """
        完整执行分而治之 CoT 推理。
        
        返回:
            包含子问题、子答案和最终答案的字典
        """
        # 第一步：拆解
        sub_questions = self.decompose(question)
        
        # 第二步：逐一求解
        sub_answers = []
        context = ""
        for sq in sub_questions:
            answer = self.solve_sub_question(sq, context)
            sub_answers.append(answer)
            context += f"\n子问题：{sq}\n答案：{answer}\n"
        
        # 第三步：汇总
        summary_prompt = (
            f"## 原始问题\n{question}\n\n"
            f"## 各子问题的答案\n"
        )
        for i, (sq, sa) in enumerate(
            zip(sub_questions, sub_answers), 1
        ):
            summary_prompt += f"\n子问题 {i}: {sq}\n答案: {sa}\n"
        summary_prompt += (
            "\n请根据以上子问题的答案，"
            "给出原始问题的最终答案。"
        )
        
        final_response = self.client.chat(
            messages=[{"role": "user", "content": summary_prompt}],
        )
        
        return {
            "question": question,
            "sub_questions": sub_questions,
            "sub_answers": sub_answers,
            "final_answer": final_response.content,
        }

技巧二：自我验证（Self-Verification）

在 CoT 推理结束后，让模型重新检查自己的推理过程。这种"回头看"的机制可以发现中间步骤的错误，显著提高最终答案的准确率。实现方式是：将原始问题和 CoT 输出一起发给模型，要求它验证推理是否正确。

自我验证在生产环境中非常实用，特别是对于高风险决策场景（如金融计算、医疗建议）。你可以设置一个验证阈值——如果模型对推理结果的置信度低于阈值，则触发二次验证或人工审核。

技巧三：多路径采样 + 投票

这是 Self-Consistency（下一章会详细讲解）的核心思想：对同一个问题执行多次 CoT（使用较高 temperature），然后对最终答案进行多数投票。如果多数推理路径得出相同答案，说明该答案更可靠。这种方式比单次 CoT 更鲁棒，但成本也相应增加。在实践中，通常采样 3-5 次即可获得较好的效果。

📌 小结

CoT 和 ToT 是 Prompt Engineering 中最核心的推理引导技术。CoT 是基础——通过逐步推理提高准确性；ToT 是进阶——通过多路径搜索提高鲁棒性。在实际项目中，建议从 CoT 开始，当准确率无法满足要求时再引入 ToT。高级技巧（分而治之、自我验证、多路径投票）可以组合使用，构建更可靠的推理系统。下一章我们将学习自洽性（Self-Consistency），它通过多次采样和投票进一步提升 CoT 的可靠性。
