3.6 结构化输出（JSON Mode、XML、Structured Outputs）

模块三：Prompt Engineering 预计学习：45 分钟

学习目标

  • 理解为什么大模型应用需要结构化输出
  • 掌握 JSON Mode 的原理与使用方法
  • 学会使用 XML 标签 强制模型输出特定格式
  • 能够构建生产级的结构化输出解析与容错引擎

一、为什么需要结构化输出？

在与大模型交互时，模型默认的输出是自然语言文本。虽然文本易于人类阅读，但对于后续的代码逻辑（如数据库写入、API 调用、前端渲染）来说，非结构化文本难以解析且极不稳定。如果模型在输出中多了一个标点符号、改变了换行习惯或使用了同义词，整个正则表达式或字符串匹配逻辑都可能失效，导致系统崩溃。

结构化输出的核心目的是建立契约：我们明确告诉模型预期的数据结构（如 JSON Schema 或 XML 格式），模型承诺严格按照该格式输出。这样，后端代码可以像调用标准 API 一样可靠地处理模型的输出结果。这不仅是构建可靠 AI 应用的基石，也是实现 Agent 自主行动（如 Function Calling）的前提。

目前主流的解决方案有三种：JSON Mode（通过 API 参数强制输出 JSON）、XML 标签包裹（在 Prompt 中要求模型用标签包裹内容）以及 Pydantic 校验（在代码端进行强类型验证和自动修复）。下面我们将逐一深入探讨。

在实际项目中，我们还会遇到一种情况：模型输出了看似合法的 JSON，但其中的字段缺失或类型错误。例如，要求模型输出一个包含 name (str), age (int) 的对象，它却只输出了 {"name": "Alice"}。这种“半结构化”的输出是系统不稳定的主要来源。因此，除了 Prompt 工程，我们在代码侧也需要引入强类型校验层，以应对模型的不完美表现。

二、JSON Mode 的原理与实践

主流 LLM 提供商（如 OpenAI、Anthropic、Google）都提供了 response_format: {"type": "json_object"} 参数。开启此模式后，模型会在输出时强制生成有效的 JSON 字符串。如果模型无法生成 JSON，API 会直接报错，而不是返回不可用的自然语言。

但是，仅仅开启 JSON Mode 是不够的。你仍然需要在 Prompt 中明确提供 JSON Schema 示例，告诉模型具体的字段名、类型和嵌套结构。否则，模型可能会生成一个格式正确但内容不符合预期的 JSON（例如输出 {"result": "ok"} 而不是你需要的详细数据）。

值得注意的是，JSON Mode 通常会对模型生成过程中的采样逻辑进行约束，这可能会轻微影响模型的创造性或推理能力。因此，对于需要高度创造性的任务，不建议开启 JSON Mode，而对于数据提取、分类、API 调用等任务，则强烈推荐。

"""
structured_output.py — 结构化输出解析引擎
包含 JSON 解析、XML 提取、重试修复与 Pydantic 验证
"""

import json
import re
import time
from typing import Optional, Type
from pydantic import BaseModel, ValidationError


class StructuredOutputParser:
    """
    结构化输出解析器：处理模型返回的非标准输出。
    
    核心功能：
        1. 自动清洗 JSON（去除 Markdown 标记、截断错误）
        2. XML 标签提取
        3. 简单的重试与修复机制
    """
    
    @staticmethod
    def parse_json(raw_output: str) -> Optional[dict]:
        """
        尝试从原始输出中解析 JSON。
        
        处理情况：
            - 标准 JSON
            - Markdown 包裹的 JSON (```json ... ```)
            - 包含前后缀文本的 JSON
        """
        # 1. 尝试直接解析
        try:
            return json.loads(raw_output)
        except json.JSONDecodeError:
            pass
        
        # 2. 尝试提取 Markdown 代码块
        json_match = re.search(r'```(?:json)?\s*([\s\S]*?)\s*```', raw_output)
        if json_match:
            try:
                return json.loads(json_match.group(1))
            except json.JSONDecodeError:
                pass
        
        # 3. 尝试寻找第一个 { 和最后一个 }
        start = raw_output.find('{')
        end = raw_output.rfind('}')
        if start != -1 and end != -1:
            json_str = raw_output[start : end+1]
            try:
                return json.loads(json_str)
            except json.JSONDecodeError:
                pass
        
        return None  # 解析失败

三、XML 标签包裹法

对于不支持 JSON Mode 的模型，或者需要输出混合内容（部分自然语言，部分结构化数据）的场景，使用 XML 标签是最稳健的方法。通过要求模型使用特定的标签（如 <answer>...</answer>）包裹关键信息，我们可以利用正则表达式精准提取内容。

XML 标签法的优势在于它不依赖于模型的 JSON 生成能力。即使模型在标签外输出了一大堆解释性文字，我们仍然可以准确抓取标签内的核心数据。这种方法在构建 Agent 时尤为常用，因为 Agent 经常需要在思考过程（自然语言）和执行动作（结构化工具调用）之间切换。

在使用 XML 标签时，标签的选择也有讲究。建议使用语义明确的标签名，如 <thought>, <action>, <result> 等。这不仅让模型更容易理解你的意图，也让代码的可读性更强。此外，当需要处理多层嵌套结构时，虽然 XML 支持嵌套，但如果数据结构过于复杂，建议还是使用 JSON 配合嵌套的 XML 标签（即在 XML 标签内部输出 JSON 字符串）。

class XMLExtractor:
    """
    基于 XML 标签的内容提取器。
    """
    
    @staticmethod
    def extract_tag(content: str, tag_name: str) -> Optional[str]:
        """
        从内容中提取指定标签的内容。
        
        支持自闭合标签、嵌套标签的简单提取。
        """
        # 正则表达式匹配 ...
        # re.DOTALL 允许匹配跨行内容
        pattern = rf'<{tag_name}\s*>(.*?)'
        match = re.search(pattern, content, re.DOTALL)
        
        if match:
            return match.group(1).strip()
        return None
    
    @staticmethod
    def extract_all_tags(content: str) -> dict:
        """
        提取内容中所有定义的标签。
        返回 {tag_name: content} 字典。
        """
        # 匹配所有标签
        tags = re.findall(r'<(\w+)>(.*?)', content, re.DOTALL)
        result = {}
        for tag_name, tag_content in tags:
            result[tag_name] = tag_content.strip()
        return result


# 使用示例
raw_response = """
好的，我来分析一下这个问题。
首先，根据用户提供的信息，我们可以确定以下几点：

用户想要了解如何解析结构化数据。
核心难点在于处理模型的随机输出。


这是我的最终答案：

使用正则表达式或 JSON 解析库。

"""

extractor = XMLExtractor()
analysis = extractor.extract_tag(raw_response, "analysis")
print(f"分析内容:\n{analysis}")
# 输出: 用户想要了解如何解析结构化数据。\n核心难点在于处理模型的随机输出。
💡 最佳实践

在使用 XML 标签时，建议在 Prompt 中明确告知模型：“请将你的最终结论包裹在 <answer> 标签中”。同时，为了保险起见，可以要求模型在标签内只输出核心内容，不要包含解释性文字，这样可以进一步降低解析失败的概率。

四、Pydantic 与强类型校验

在 Python 生态中，Pydantic 是处理结构化数据的事实标准。结合大模型使用，我们可以实现“生成-校验-自动修复”**的闭环。当模型输出的 JSON 不符合预期 Schema 时（例如字段缺失、类型错误），我们可以捕获异常，并将错误信息反馈给模型，要求它重新生成，直到满足要求为止。

这种机制在生产环境中至关重要，因为它保证了下游业务逻辑接收到的数据永远是类型安全、格式正确的。这也是 LangChain 等框架底层实现 Structured Output 的核心原理。

class PydanticValidator:
    """
    基于 Pydantic 的强类型校验与自动修复引擎。
    """
    
    def __init__(self, schema: Type[BaseModel], max_retries: int = 3):
        """
        参数:
            schema: Pydantic 模型类
            max_retries: 最大重试修复次数
        """
        self.schema = schema
        self.max_retries = max_retries
    
    def validate_and_fix(self, data: dict) -> dict:
        """
        校验并尝试修复数据。
        
        返回:
            符合 Schema 的字典数据
        异常:
            ValidationError: 如果多次修复仍失败
        """
        try:
            # 1. 尝试直接校验
            instance = self.schema(**data)
            return instance.model_dump()
        except ValidationError as e:
            print(f"校验失败，开始自动修复... 错误: {e.errors()[0]['msg']}")
            # 在实际应用中，这里会调用 LLM 进行修复
            # 我们在此处仅做日志记录，
            # 真正的修复需要构建包含错误信息的 Prompt 并重试
            raise
    
    def get_schema_prompt(self) -> str:
        """生成用于 Prompt 的 JSON Schema 字符串"""
        return self.schema.model_json_schema()


# 定义数据模型
class UserResponse(BaseModel):
    name: str
    age: int
    is_active: bool = True
    tags: list[str]

# 初始化校验器
validator = PydanticValidator(schema=UserResponse)

# 模拟模型输出（缺少 tags 字段）
bad_output = {"name": "Alice", "age": 25}

try:
    result = validator.validate_and_fix(bad_output)
    print(f"校验通过: {result}")
except ValidationError as e:
    print(f"校验最终失败: {e}")
    # 下一步：构建 "修复 Prompt" -> 调用 LLM -> 再次校验

五、三种方案对比

不同的场景适合不同的结构化策略。以下表格帮助你做出选择：

维度JSON ModeXML 标签Pydantic 校验
适用场景纯数据返回，API 支持混合内容，通用场景高可靠性要求，复杂结构
模型支持OpenAI, Anthropic 等所有模型所有模型（后端实现）
稳定性高中高最高（带重试修复）
Token 消耗低中（标签占位）高（重试机制）
实现复杂度极低（API 参数）低（正则提取）高（Schema 定义 + 循环）
容错能力无（API 报错）中（正则匹配）强（类型检查 + 反馈）
⚠️ 常见陷阱

不要假设模型总是输出完美 JSON。即使开启了 JSON Mode，小概率事件下模型仍可能截断输出或格式错误。务必在代码中编写健壮的解析逻辑（如本节的 parse_json 函数），而不是简单粗暴地 json.loads(response.text)。

六、生成修复 Prompt

当结构化校验失败时，我们需要构建一个“修复 Prompt"让模型修正错误。这个 Prompt 必须包含原始输入、预期 Schema以及具体的错误信息。

def generate_fix_prompt(
    original_input: str,
    bad_output: str,
    schema_desc: str,
    error_msg: str
) -> str:
    """
    生成用于修复结构化输出的 Prompt。
    
    参数:
        original_input: 用户的原始问题
        bad_output: 模型生成的错误输出
        schema_desc: JSON Schema 描述
        error_msg: 具体的校验错误信息
    返回:
        修复 Prompt 字符串
    """
    return f"""
# 任务
你之前的输出不符合预期的 JSON 格式。请根据错误信息修正你的回答。

# 原始问题
{original_input}

# 你的错误输出
{bad_output}

# 预期的 JSON Schema
{schema_desc}

# 校验错误
{error_msg}

# 要求
1. 严格遵循 JSON Schema 格式
2. 不要包含任何解释性文字，只输出 JSON
3. 确保所有必填字段都存在且类型正确
"""

# 使用示例
fix_prompt = generate_fix_prompt(
    original_input="分析这段文本的情感",
    bad_output='{"sentiment": "positive"}', # 假设 Schema 要求包含 'confidence'
    schema_desc='{"type": "object", "properties": {"sentiment": {"type": "string"}, "confidence": {"type": "number"}}}',
    error_msg="'confidence' is a required property"
)
print(fix_prompt)

七、自动重试循环实现

在生产环境中，我们通常会将校验、修复和重试封装在一个循环中。这样可以实现“生成-校验-修复”的闭环，确保最终返回给业务逻辑的数据一定是合法的。

class RetryLoop:
    """
    自动重试循环：直到输出符合 Schema 或达到最大重试次数。
    
    工作流程：
        1. 调用 LLM 获取输出
        2. 解析 JSON
        3. 使用 Pydantic 校验
        4. 如果失败，构建修复 Prompt -> 再次调用 LLM
        5. 重复直到成功或超时
    """
    
    def __init__(self, llm_client, schema, max_retries=3):
        self.client = llm_client
        self.schema = schema
        self.max_retries = max_retries
        self.validator = PydanticValidator(schema=schema)
    
    def run(self, original_input: str) -> dict:
        """
        执行重试循环。
        
        参数:
            original_input: 用户的原始问题
        返回:
            符合 Schema 的字典数据
        """
        # 1. 初始 Prompt
        prompt = f"请回答以下问题，并以 JSON 格式返回结果。\nSchema: {self.validator.get_schema_prompt()}\n\n问题：{original_input}"
        
        current_output = ""
        
        for attempt in range(self.max_retries):
            try:
                # 调用 LLM
                response = self.client.chat(
                    messages=[{"role": "user", "content": prompt}]
                )
                current_output = response.content
                
                # 解析 JSON
                data = StructuredOutputParser.parse_json(current_output)
                if not data:
                    raise ValueError("无法解析 JSON")
                
                # 校验
                return self.validator.validate_and_fix(data)
                
            except (ValueError, ValidationError) as e:
                print(f"第 {attempt+1} 次尝试失败: {e}")
                
                # 如果是最后一次，抛出异常
                if attempt == self.max_retries - 1:
                    raise Exception(f"达到最大重试次数，最终输出: {current_output}")
                
                # 构建修复 Prompt
                prompt = generate_fix_prompt(
                    original_input=original_input,
                    bad_output=current_output,
                    schema_desc=self.validator.get_schema_prompt(),
                    error_msg=str(e)
                )
                print("构建修复 Prompt，准备重试...")
        
        raise Exception("逻辑不应到达此处")

📌 小结

结构化输出是将大模型接入生产系统的必经之路。JSON Mode 是最便捷的方式，XML 标签 提供了极高的通用性，而 Pydantic 校验 则是保证数据质量的最后一道防线。在实际开发中，建议结合使用：使用 JSON Mode 或 XML 提示模型输出格式，并在后端使用 Pydantic 进行强类型验证和自动重试修复。下一章我们将学习Prompt 模板管理与版本控制，探讨如何在工程中高效维护这些复杂的 Prompt。
