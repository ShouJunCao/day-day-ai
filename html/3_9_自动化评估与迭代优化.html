3.9 自动化评估与迭代优化

在 Prompt 工程的生命周期中，"写出来"只是第一步，"写得好不好"必须依靠客观评估。缺乏系统化评估的 Prompt 迭代，就像没有单元测试的代码开发——短期内可能跑通 demo，但一旦投入生产环境，面对真实用户的多样化输入，性能往往会断崖式下跌。

本章将构建一套生产级 自动化评估管道（Evaluation Pipeline），融合 LLM-as-a-Judge、规则引擎与回归检测，帮助你从"凭感觉改 Prompt"走向"数据驱动迭代"。

为什么需要自动化评估？

传统的人工抽查方式存在三个致命缺陷：样本量小（通常只看 10-20 条）、主观偏差大（不同标注者标准不一）、无法快速反馈（改一次 Prompt 等半天评估结果）。在 CI/CD 流水线中，每次代码提交都会自动运行测试用例；Prompt 的迭代同样需要建立类似的 Prompt CI 机制：每次修改后，自动在黄金测试集上跑分，分数不降才允许合并。

评估指标体系

一个完善的 Prompt 评估体系应覆盖四个维度：准确性（答案是否正确）、一致性（多次运行是否稳定）、安全性（是否输出有害内容）和 延迟（响应时间是否可接受）。

LLM-as-a-Judge 原理与实践

LLM-as-a-Judge 的核心思想是：用一个更强的模型（如 GPT-4o、Claude 3.5 Sonnet）充当"考官"，对被测 Prompt 的输出进行打分。相较于 BLEU/ROUGE 等基于词重叠的传统指标，LLM 评判能真正理解语义意图，对代码生成、创意写作等开放域任务效果显著。

但 LLM-as-a-Judge 也有明显局限：位置偏差（先看的答案容易打高分）、自利偏差（同一模型给自己打分偏高）、verbosity 偏好（越长越觉得好）。因此，必须配合 规则引擎 做交叉验证，避免单一评估器的盲区。

实战：构建混合评估管道

以下实现了一个生产级的评估管道，支持 LLM 裁判与规则引擎的并行评分、结果聚合与回归检测。

评估器接口与管道定义

# prompt_evaluator.py
from __future__ import annotations
import asyncio
import logging
from dataclasses import dataclass, field
from typing import Protocol, Sequence
import httpx

logger = logging.getLogger(__name__)

class ScoreMetric(Protocol):
    """评估指标接口"""
    async def score(self, prompt: str, output: str, reference: str | None) -> float: ...

@dataclass(frozen=True)
class EvaluationResult:
    prompt_id: str
    metrics: dict[str, float]
    latency_ms: float
    judge_feedback: str | None = None

LLM-as-a-Judge 实现

@dataclass
class LLMEvaluator:
    """使用 LLM 作为评估器"""
    judge_model: str
    api_key: str
    base_url: str = "https://api.openai.com/v1"
    rubric: str = "评估回答的准确性、完整性和逻辑性（0-100分）"
    client: httpx.AsyncClient | None = field(default=None, repr=False)
    _timeout: float = 30.0

    def __post_init__(self) -> None:
        if self.client is None:
            self.client = httpx.AsyncClient(timeout=self._timeout)

    async def _call_judge(self, prompt: str, output: str) -> str:
        """调用裁判模型"""
        if not self.client:
            raise RuntimeError("HTTP client is not initialized")
        payload = {
            "model": self.judge_model,
            "messages": [
                {"role": "system", "content": f"你是一个严格的评估专家。{self.rubric}。只返回数字分数。"},
                {"role": "user", "content": f"Prompt:\n{prompt}\n---\nOutput:\n{output}"}
            ]
        }
        try:
            resp = await self.client.post(
                f"{self.base_url}/chat/completions",
                json=payload,
                headers={"Authorization": f"Bearer {self.api_key}"}
            )
            resp.raise_for_status()
            return resp.json()["choices"][0]["message"]["content"].strip()
        except Exception as exc:
            logger.error("Judge API call failed: %s", exc)
            return "0"

    async def score(self, prompt: str, output: str, reference: str | None = None) -> float:
        raw = await self._call_judge(prompt, output)
        try:
            return max(0.0, min(100.0, float(raw)))
        except ValueError:
            logger.warning("Non-numeric judge score: %s", raw)
            return 0.0

规则引擎与聚合逻辑

@dataclass
class RuleBasedEvaluator:
    """基于规则的轻量评估器"""
    min_length: int = 10
    max_length: int = 2000
    forbidden_terms: Sequence[str] = field(default_factory=lambda: ["不知道", "无法回答", "抱歉"])

    async def score(self, prompt: str, output: str, reference: str | None = None) -> float:
        if not output:
            return 0.0
        length_penalty = 1.0 if self.min_length <= len(output) <= self.max_length else 0.5
        term_penalty = sum(1 for t in self.forbidden_terms if t in output) * 10.0
        return max(0.0, 100.0 - term_penalty) * length_penalty

批量评估与回归检测

单次评估意义有限，必须对黄金测试集进行批量跑分，并计算均值、标准差、P5/P95 分位数。更重要的是 回归检测：每次修改 Prompt 后，对比当前分数与历史基线，如果核心指标下降超过阈值（如 5 分），则自动触发回滚。

# metrics.py - 回归检测核心逻辑
def detect_regressions(
    current: list[MetricSummary],
    baseline: dict[str, float],
    threshold: float = 5.0
) -> EvaluationSummary:
    regressions: list[str] = []
    for summary in current:
        base_score = baseline.get(summary.metric_name)
        if base_score is not None:
            if summary.mean < (base_score - threshold):
                regressions.append(summary.metric_name)
    return EvaluationSummary(
        summaries=current,
        regression_detected=bool(regressions),
        regression_metrics=regressions
    )
💡 最佳实践：黄金测试集维护

黄金测试集不是一次性产物，需要持续维护。每当发现新的 bad case，就将其加入测试集；当业务规则变化，及时更新参考答案。测试集应覆盖：常规场景（60%）、边界条件（20%）、对抗输入（10%）、多语言/多格式（10%）。

高级技巧：降低 Judge 偏差

为提升 LLM-as-a-Judge 的可靠性，可采用以下策略：

  • 双盲评审：将候选答案与参考答案的顺序随机打乱，避免位置偏差。
  • 多模型投票：使用 2-3 个不同模型的打分取平均，降低单一模型的系统偏差。
  • 结构化打分：要求 Judge 按维度（准确性、完整性、安全性）分别打分，而不是只给一个总分。
  • 温度归零：调用 Judge 模型时设置 temperature=0，确保评分的一致性。
⚠️ 常见陷阱
  • 过度依赖单一指标：只看平均分，忽略长尾分布。P5 分位数的退化往往意味着某些特定场景的严重故障。
  • 基线漂移：随着业务演进，旧的基线可能不再适用。应定期（如每月）重新校准基线分数。
  • 评估成本失控：LLM-as-a-Judge 的 API 调用成本可能很高。建议对高频测试用例缓存 Judge 结果，或先用规则引擎做快速筛选。

本章小结

自动化评估是 Prompt 工程从"手工作坊"走向"工业化生产"的关键基础设施。通过构建混合评估管道、建立黄金测试集、实施回归检测，你可以确保每次 Prompt 迭代都朝着正确的方向演进，而不是在黑暗中摸索。
