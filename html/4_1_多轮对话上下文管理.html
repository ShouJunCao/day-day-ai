4.1 多轮对话上下文管理

滑动窗口 (Sliding Window) · 摘要压缩 (Summarization) · Token 预算控制

📌 核心要点：大模型虽然支持 128K 甚至 200K 的长上下文，但"能用"不代表"好用"。随着上下文增长，API 成本指数上升，且模型易出现"中间丢失"现象。本章介绍三种工程化策略：
1. 滑动窗口：丢弃旧消息，只保留最新对话。
2. 摘要压缩：用 LLM 总结历史对话，替换为一条 Summary 消息。
3. Token 预算：严格控制输入给模型的 Token 总量，防止超出上下文限制。

为什么需要上下文管理？

在生产环境中，用户的对话可能持续数小时甚至数天。如果将历史记录全部塞入 Prompt，不仅会产生高昂的 Token 费用，还会导致模型注意力分散，响应延迟增加。更严重的是，当对话超过模型的上下文窗口限制时，API 会直接返回 Context Length Exceeded 错误，导致服务崩溃。

策略一：滑动窗口 (Sliding Window)

最简单高效的策略。设定一个最大窗口大小（如保留最近 10 条消息），当新消息加入导致超过限制时，自动丢弃最旧的历史记录。

⚠️ 注意：滑动窗口的缺点是会永久丢失被裁剪掉的历史信息。如果用户在第 1 条消息中提到了重要背景（如"我在开发一个电商系统"），窗口滑动后这个信息就丢失了。

实战代码：滑动窗口管理器

以下实现了一个生产级的滑动上下文管理器，支持 System Prompt 保护和自动 Token 截断。

# context_window.py
from dataclasses import dataclass, field

class TokenEstimator:
    """Token 估算器 - 生产环境推荐替换为 tiktoken"""

    @staticmethod
    def estimate(text: str) -> int:
        """粗略估算 Token 数量"""
        if not text:
            return 0
        cn_count = sum(1 for c in text if ord(c) > 127)
        en_count = len(text) - cn_count
        return max(1, int(cn_count / 1.5) + int(en_count / 4))

@dataclass(frozen=True)
class Message:
    """不可变消息对象"""
    role: str
    content: str
    token_count: int = 0

    def __post_init__(self):
        if self.token_count == 0:
            object.__setattr__(self, "token_count",
                TokenEstimator.estimate(self.content))

@dataclass
class SlidingWindowContext:
    """滑动窗口管理器"""
    max_turns: int = 10
    max_tokens: int = 4000
    messages: list[Message] = field(default_factory=list)

    def add(self, role: str, content: str) -> None:
        """添加新消息并触发自动裁剪"""
        msg = Message(role=role, content=content)
        self.messages.append(msg)
        self._trim()

    def _trim(self) -> None:
        # 优先级 1: Token 预算检查
        while self._current_tokens > self.max_tokens and len(self.messages) > 2:
            self._pop_oldest()
        # 优先级 2: Turn 数量检查
        while len(self.messages) > self.max_turns:
            self._pop_oldest()

    def _pop_oldest(self) -> None:
        """删除最旧的非 System 消息"""
        if not self.messages:
            return
        if self.messages[0].role == "system" and len(self.messages) > 1:
            self.messages.pop(1)
        elif self.messages[0].role != "system":
            self.messages.pop(0)

    @property
    def _current_tokens(self) -> int:
        return sum(m.token_count for m in self.messages)

    def get_context(self) -> list[dict[str, str]]:
        return [{"role": m.role, "content": m.content} for m in self.messages]

策略二：摘要压缩 (Summarization)

当窗口滑到一定极限时，简单丢弃会导致上下文断裂。更好的做法是调用一个廉价的小模型（如 gpt-4o-mini），将旧对话压缩成一段摘要（Summary），替换回 System Prompt 或放在历史开头。

实战代码：摘要压缩器

# summary_compressor.py
import os
import httpx
from dataclasses import dataclass

@dataclass
class CompressorConfig:
    api_key: str = os.getenv("API_KEY", "")
    base_url: str = "https://api.openai.com/v1"
    model: str = "gpt-4o-mini"
    max_summary_tokens: int = 150

class SummaryCompressor:
    def __init__(self, config: CompressorConfig = None):
        self.config = config or CompressorConfig()

    async def compress(self, old_messages: list[dict]) -> str:
        """将旧对话压缩为摘要"""
        history_text = "\n".join(
            f"[{m.get('role')}] {m.get('content')}" for m in old_messages
        )
        async with httpx.AsyncClient(timeout=30) as client:
            resp = await client.post(
                f"{self.config.base_url}/chat/completions",
                json={
                    "model": self.config.model,
                    "messages": [
                        {"role": "system", "content":
                         "请将以下对话压缩为150字以内的中文摘要。"},
                        {"role": "user", "content": history_text},
                    ],
                    "max_tokens": self.config.max_summary_tokens,
                },
                headers={
                    "Authorization": f"Bearer {self.config.api_key}",
                    "Content-Type": "application/json",
                },
            )
            resp.raise_for_status()
            return resp.json()["choices"][0]["message"]["content"]

    def build_context_with_summary(
        self, summary: str, recent: list[dict]
    ) -> list[dict]:
        """摘要 + 最近消息 = 完整上下文"""
        return [
            {"role": "system", "content": f"之前对话的摘要:\n{summary}"},
            *recent,
        ]

策略三：Token 预算控制 (Token Budget)

在发送请求前，必须精确计算即将消耗的 Token。如果超过模型上限（如 8K、32K），需立即触发截断或拒绝服务，避免 API 报错。

💡 生产建议：实际项目中建议使用 tiktoken 库进行精确 Token 计数。此处演示的是粗略估算方法。

实战代码：Token 预算控制器

# token_budget.py
class TokenBudgetController:
    """Token 预算控制器"""

    def __init__(self, limit: int = 8192, model: str = "gpt-4o"):
        self.limit = limit
        self.model = model

    def count_tokens(self, messages: list[dict]) -> int:
        """计算消息列表的 Token 总量"""
        total = 0
        for msg in messages:
            content = msg.get("content", "")
            total += self._estimate(content)
            total += 4  # 角色 overhead
        return total + 3  # 系统 overhead

    def _estimate(self, text: str) -> int:
        """粗略估算 Token 数"""
        if not text:
            return 0
        cn_count = sum(1 for c in text if ord(c) > 127)
        en_count = len(text) - cn_count
        return int(cn_count / 1.5) + int(en_count / 4)

    def check(self, messages: list[dict]) -> dict:
        """检查 Token 预算"""
        used = self.count_tokens(messages)
        return {
            "ok": used <= self.limit,
            "used": used,
            "remaining": max(0, self.limit - used),
            "over_by": max(0, used - self.limit),
        }

    def trim_to_fit(self, messages: list[dict]) -> list[dict]:
        """自动裁剪消息直到符合预算"""
        result = list(messages)
        while self.count_tokens(result) > self.limit and len(result) > 1:
            idx = 1 if result[0].get("role") == "system" else 0
            result.pop(idx)
        return result

策略对比总结

策略实现复杂度成本上下文丢失风险适用场景
滑动窗口⭐ 低💰 低 🔴 高（直接丢弃）简单问答、短期任务
摘要压缩⭐⭐ 中💰💰 中 🟢 低（保留语义）长期陪伴、复杂任务
Token 预算⭐ 低💰 无 🟢 可控所有生产级系统必备

最佳实践：组合使用

在实际生产系统中，推荐组合使用这三种策略：

  1. 用 Token 预算 设定硬性上限，防止 API 报错
  2. 用 滑动窗口 处理日常对话，保持上下文精简
  3. 当窗口即将满载时，触发 摘要压缩，保留关键语义
