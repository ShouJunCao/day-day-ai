5.4 向量索引算法（HNSW、IVF、PQ、ScaNN）

模块五：向量数据库与 Embedding 预计学习：50 分钟

学习目标

  • 理解向量索引的核心作用：将 O(N) 暴力搜索优化为近似 O(log N)
  • 掌握 HNSW、IVF、PQ 三种主流索引的原理和适用场景
  • 了解索引选择对检索精度、速度和内存的影响
  • 能够根据数据规模和性能要求选择合适的索引类型

一、为什么需要向量索引？

假设有 100 万个 768 维向量，要找到与查询向量最相似的 10 个。暴力搜索（Flat 索引）的方法是：逐个计算查询向量与每个向量的距离，然后排序取前 10 个。这意味着 100 万次 768 维向量的距离计算——在普通 CPU 上需要几百毫秒甚至数秒。

对于需要实时响应的在线服务（如 RAG 系统的检索阶段），这个延迟是不可接受的。向量索引的核心价值就是用空间换时间——在写入阶段预建索引结构，使得搜索时的计算量从 O(N) 降到 O(log N) 甚至更低。

不同类型的索引在精度、速度和内存开销之间有不同的权衡。Flat 索引虽然精确但只适合小规模数据；HNSW 速度快精度高但内存占用大；IVF 通过聚类缩小搜索范围，内存效率好但需要训练；PQ 以量化压缩为代价换取极致的存储效率。没有一种索引在所有场景下都是最优的，理解各索引的原理和特点是做出正确选型的前提。

以一个具体的例子来说明索引的价值：假设你有一个包含 1000 万篇文档的向量库，每篇文档对应一个 768 维向量。如果用暴力搜索，每次查询需要计算 1000 万次距离，耗时可能在 1-3 秒之间。但如果使用 HNSW 索引，同样的查询只需检查几千个节点，耗时降到 10-50 毫秒——提速 20-100 倍。这就是为什么向量数据库必须依赖索引结构。

二、HNSW 索引

HNSW（Hierarchical Navigable Small World，分层可导航小世界图）是目前最流行的向量索引算法。它构建了一个多层图结构，顶层节点少、底层节点多。搜索时从顶层开始，找到最近节点后下沉到下一层，逐层逼近，最终在底层找到最相似的 K 个向量。

这种"从粗到细"的策略非常高效。想象你在一个陌生城市找人——先坐地铁到正确的区（顶层粗搜索），再步行到具体的街道（底层精搜索），而不是一开始就从第一条街走到最后一条街。HNSW 的多层图就是这个道理，它将搜索复杂度从 O(N) 降到了 O(log N)。

HNSW 的优势非常明显：速度极快（近似 O(log N) 复杂度），精度极高（接近 100% 召回率），且适合各种数据规模。它是 Milvus、Qdrant、Chroma 等主流向量数据库的默认索引类型。

但 HNSW 的代价是内存占用大。每个节点需要存储与多个邻居的连接信息，在大规模场景下，图结构的内存开销可能远超原始向量本身。100 万个 768 维向量，纯向量数据约 3 GB，但 HNSW 索引可能需要 5-10 GB。

HNSW 有两个关键参数：M（每个节点的最大连接数）和efConstruction（构建时的搜索宽度）。M 越大，图越密集，搜索越快但内存越大；efConstruction 越大，索引质量越高但构建越慢。实际应用中，M=16、efConstruction=200 是一个不错的起点。

"""
vector_indexes.py — 向量索引算法实现
包含 HNSW、IVF、PQ 三种主流索引的教学实现。
"""
import math
import random
from typing import Optional
from dataclasses import dataclass, field


class HNSWIndex:
    """
    HNSW 索引的简化实现。

    原理：
    - 构建多层图结构，顶层节点少、底层节点多
    - 搜索时从顶层开始，找到最近节点后下沉到下一层
    - 逐层逼近，最终在底层找到最相似的 K 个节点
    """

    def __init__(self, dimension: int, M: int = 16, max_layer: int = 4):
        """
        初始化 HNSW 索引。

        参数：
        - dimension (int): 向量维度
        - M (int): 每层最大连接数
        - max_layer (int): 最大层数
        """
        self.dimension = dimension
        self.M = M
        self.max_layer = max_layer
        self.nodes: dict[str, HNSWNode] = {}
        self.entry_point: Optional[str] = None

    def insert(self, vector: list[float], node_id: str) -> None:
        """插入向量节点"""
        node_layer = self._random_layer()
        node = HNSWNode(vector=vector, node_id=node_id, layer=node_layer)
        self.nodes[node_id] = node

        if self.entry_point is None:
            self.entry_point = node_id
            return

        # 将新节点连接到每层距离最近的 M 个已有节点
        for layer in range(node_layer + 1):
            existing = [nid for nid, n in self.nodes.items()
                       if n.layer >= layer and nid != node_id]
            existing.sort(key=lambda nid: euclidean_distance(
                self.nodes[nid].vector, vector))
            nearest = existing[:self.M]
            for nid in nearest:
                if layer < len(self.nodes[nid].neighbors):
                    if node_id not in self.nodes[nid].neighbors[layer]:
                        self.nodes[nid].neighbors[layer].append(node_id)
                if layer < len(node.neighbors):
                    if nid not in node.neighbors[layer]:
                        node.neighbors[layer].append(nid)

    def search(self, query_vector: list[float], top_k: int = 5) -> list[tuple[str, float]]:
        """搜索最相似的 K 个向量"""
        if not self.nodes:
            return []

        current_id = self.entry_point

        # 从顶层逐层下沉到底层
        for layer in range(self.max_layer, -1, -1):
            changed = True
            while changed:
                changed = False
                current_node = self.nodes.get(current_id)
                if not current_node or layer >= len(current_node.neighbors):
                    break
                for neighbor_id in current_node.neighbors[layer]:
                    neighbor = self.nodes.get(neighbor_id)
                    if not neighbor:
                        continue
                    curr_dist = euclidean_distance(
                        self.nodes[current_id].vector, query_vector)
                    nb_dist = euclidean_distance(
                        neighbor.vector, query_vector)
                    if nb_dist < curr_dist:
                        current_id = neighbor_id
                        changed = True

        # 在底层找 top_k
        candidates = {current_id}
        bottom = self.nodes.get(current_id)
        if bottom and bottom.neighbors:
            candidates.update(bottom.neighbors[0])

        results = []
        for nid in candidates:
            node = self.nodes.get(nid)
            if node:
                dist = euclidean_distance(node.vector, query_vector)
                results.append((nid, dist))

        results.sort(key=lambda x: x[1])
        return results[:top_k]

三、IVF 索引

IVF（Inverted File Index，倒排文件索引）的思路更直观：先用 K-Means 将所有向量聚类为 nlist 个簇（比如 100 个），每个向量归入最近的簇。搜索时，先找到查询向量最近的 nprobe 个簇（比如 5 个），只在这些簇中暴力搜索。

这种"先缩小范围再精确查找"的策略大幅减少了计算量。100 万个向量分成 100 个簇，每个簇平均 1 万个向量。如果只探查 5 个簇，搜索量就从 100 万降到了 5 万——减少 95% 的计算。

IVF 的优势是内存占用小——只需要存储聚类中心和倒排表，不需要像 HNSW 那样维护复杂的图结构。对于数据量在百万到亿级的场景，IVF 是性价比很高的选择。

但 IVF 有两个需要注意的问题：需要训练阶段（K-Means 聚类需要遍历全量数据），且精度略低于 HNSW（因为搜索范围被限制在少数簇中，如果查询向量的最近邻恰好不在探查的簇里，就会漏掉结果）。nprobe 参数就是精度和速度之间的权衡——nprobe 越大，精度越高但速度越慢。

class IVFIndex:
    """
    IVF 索引的简化实现。

    原理：
    1. 训练：用 K-Means 将所有向量聚类为 nlist 个簇
    2. 索引：每个向量归入最近的簇，建立倒排表
    3. 搜索：找到查询最近的 nprobe 个簇，只在簇中搜索
    """

    def __init__(self, dimension: int, nlist: int = 100, nprobe: int = 10):
        self.dimension = dimension
        self.nlist = nlist
        self.nprobe = nprobe
        self.centroids: list[list[float]] = []
        self.inverted_lists: dict[int, list[str]] = {}
        self.nodes: dict[str, list[float]] = {}
        self.is_trained = False

    def train(self, vectors: list[list[float]]) -> None:
        """训练：K-Means 聚类"""
        random.seed(42)
        self.centroids = random.sample(vectors, min(self.nlist, len(vectors)))
        for vec in vectors:
            dists = [euclidean_distance(vec, c) for c in self.centroids]
            cluster_id = dists.index(min(dists))
            self.inverted_lists.setdefault(cluster_id, [])
        self.is_trained = True

    def insert(self, vector: list[float], node_id: str) -> None:
        """插入到倒排表"""
        self.nodes[node_id] = vector
        if not self.is_trained:
            return
        dists = [euclidean_distance(vector, c) for c in self.centroids]
        cluster_id = dists.index(min(dists))
        self.inverted_lists.setdefault(cluster_id, []).append(node_id)

    def search(self, query_vector: list[float], top_k: int = 5) -> list[tuple[str, float]]:
        """搜索：只在 nprobe 个最近簇中查找"""
        centroid_dists = [euclidean_distance(query_vector, c) for c in self.centroids]
        sorted_clusters = sorted(range(len(centroid_dists)), key=lambda i: centroid_dists[i])
        probe_clusters = sorted_clusters[:self.nprobe]

        candidates = set()
        for cid in probe_clusters:
            if cid in self.inverted_lists:
                candidates.update(self.inverted_lists[cid])

        results = [(nid, euclidean_distance(self.nodes[nid], query_vector))
                   for nid in candidates if nid in self.nodes]
        results.sort(key=lambda x: x[1])
        return results[:top_k]

四、PQ 索引

PQ（Product Quantization，乘积量化）的出发点完全不同——它不追求精度，而是追求极致的内存压缩。

PQ 的做法是将 D 维向量切分为 m 段，每段 D/m 维。对每段独立运行 K-Means，生成 k 个聚类中心（称为码本）。每个向量不再存储原始浮点数，而是存储 m 个码字 ID（每个 ID 占 1 字节）。以 768 维 float32 向量为例：原始大小 768 × 4 = 3072 字节，用 PQ（m=48, k=256）压缩后只需 48 字节，压缩比高达 64 倍——100 万个向量从 3 GB 压缩到 48 MB。

PQ 的代价是量化误差——用码本中心近似原始向量必然引入精度损失。在实际应用中，PQ 的召回率通常比 HNSW 低 5-15 个百分点，但在超大规模场景下，这是唯一可行的方案。

class PQIndex:
    """
    PQ（乘积量化）索引的简化实现。

    原理：
    1. 将 D 维向量分成 m 段，每段 D/m 维
    2. 对每段独立运行 K-Means，生成 k 个码本中心
    3. 每个向量用 m 个码字 ID 表示（压缩存储）
    4. 搜索时用预计算距离表加速（ADC 方法）
    """

    def __init__(self, dimension: int, m: int = 4, k: int = 256):
        self.dimension = dimension
        self.m = m
        self.k = k
        self.sub_dim = dimension // m
        self.codebooks: list[list[list[float]]] = []
        self.encoded: dict[str, list[int]] = {}
        self.nodes: dict[str, list[float]] = {}
        self.is_trained = False

    def train(self, vectors: list[list[float]]) -> None:
        """训练：对每段运行 K-Means"""
        random.seed(42)
        self.codebooks = []
        for seg_idx in range(self.m):
            sub_vectors = [v[seg_idx * self.sub_dim: (seg_idx + 1) * self.sub_dim] for v in vectors]
            centers = random.sample(sub_vectors, min(self.k, len(sub_vectors)))
            self.codebooks.append(centers)
        self.is_trained = True

    def encode(self, vector: list[float]) -> list[int]:
        """将向量编码为码字 ID 列表"""
        sub_vectors = [vector[i * self.sub_dim: (i + 1) * self.sub_dim] for i in range(self.m)]
        codes = []
        for seg_idx, sub_vec in enumerate(sub_vectors):
            dists = [euclidean_distance(sub_vec, c) for c in self.codebooks[seg_idx]]
            codes.append(dists.index(min(dists)))
        return codes

    def search(self, query_vector: list[float], top_k: int = 5) -> list[tuple[str, float]]:
        """用预计算距离表加速搜索（ADC）"""
        query_subs = [query_vector[i * self.sub_dim: (i + 1) * self.sub_dim] for i in range(self.m)]
        distance_table = []
        for seg_idx, qsub in enumerate(query_subs):
            table = [euclidean_distance(qsub, c) for c in self.codebooks[seg_idx]]
            distance_table.append(table)

        results = []
        for node_id, codes in self.encoded.items():
            approx_dist = sum(distance_table[i][codes[i]] for i in range(self.m))
            results.append((node_id, approx_dist))
        results.sort(key=lambda x: x[1])
        return results[:top_k]

六、ScaNN 简介

ScaNN（Scalable Nearest Neighbors）是 Google 提出的向量搜索算法，在 Yandex 的 ANN-benchmarks 上多次获得速度和精度的双料第一。它结合了 ANH（Anisotropic Vector Quantization）和 SQ（Squared Q-Index）两种技术，核心思想是将搜索分为"粗筛 + 精排"两阶段。

第一阶段用 PQ 快速筛选候选集（内存小、速度快但有量化误差），第二阶段用更精确的方法（如基于树或暴力搜索）在候选集中精排。这种两阶段策略在超大规模场景下比单纯的 HNSW 或 IVF 更高效，因为大多数向量在第一阶段就被排除了，不需要参与第二阶段的精确计算。

💡 索引选择建议

如果你不确定选哪个索引：先用 HNSW。它在精度和速度之间取得了最好的平衡，是大多数场景下的默认选择。只有当内存成为瓶颈时（向量规模超过一亿），才考虑 PQ 量化来压缩存储，或 IVF 来平衡精度和内存。

七、小结

向量索引是向量数据库的核心引擎，决定了检索的速度、精度和资源开销。Flat 索引简单精确但只适合小规模数据；HNSW 速度快、精度高但内存开销大，是最通用的选择；IVF 通过聚类减少搜索范围，内存效率好但需要训练阶段；PQ 以量化压缩为代价换取极致的存储效率，适合超大规模场景。在实际项目中，建议先用 HNSW 验证方案，根据实际数据量和性能表现决定是否切换到其他索引。

混合索引也是一个值得关注的方向。例如 Milvus 支持 IVF+PQ 组合索引——先用 IVF 将搜索范围缩小到少数簇，再用 PQ 在每个簇中快速计算近似距离。这种组合策略可以同时获得 IVF 的范围缩减优势和 PQ 的内存压缩优势，在精度和效率之间取得更好的平衡。

此外，ScaNN 等新型算法通过两阶段策略（粗筛 + 精排）在超大规模场景下表现优异，是值得关注的新趋势。

在下一节 5.5 中，我们将深入学习相似度度量的具体计算方法和选择策略。

最后提醒一点：索引参数调优是持续的过程。随着数据量增长和查询模式变化，需要定期评估索引的召回率和延迟，必要时重新训练或重建索引。
