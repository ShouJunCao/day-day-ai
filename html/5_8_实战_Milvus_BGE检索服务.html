学习目标

  • 完成从环境搭建到检索服务的全流程
  • 掌握 Milvus 连接、集合创建、索引构建的完整操作
  • 学会使用 BGE-M3 本地模型生成向量并写入 Milvus

一、环境准备

本节实战需要安装三个核心依赖：FlagEmbedding（BGE-M3 模型）、pymilvus（Milvus 客户端）和 torch（深度学习框架）。建议使用虚拟环境隔离依赖，避免版本冲突。

📦 环境要求
  • Python：3.9+
  • GPU：推荐（BGE-M3 需要 GPU 加速推理，CPU 也可以但慢）
  • 显存：至少 4 GB（FP16 模式下）
  • 磁盘：BGE-M3 模型约 2.2 GB

如果你的机器没有 GPU，也可以使用 CPU 推理，只是速度会慢很多。BGE-M3 在 CPU 上处理一段 100 字的文本大约需要 0.5-2 秒，而在 GPU 上只需 20-50 毫秒。对于批量索引场景，CPU 推理可能成为性能瓶颈，建议至少使用支持 CUDA 的 GPU。

此外，建议在 Linux 或 macOS 环境下进行实验。Windows 环境下可能会遇到 CUDA 驱动兼容性问题，如果必须使用 Windows，建议通过 WSL2 运行。

# 创建虚拟环境
python3 -m venv .venv
source .venv/bin/activate

# 安装依赖
pip install FlagEmbedding torch pymilvus

# 如果使用 Milvus 服务器版（非 MilvusLite），还需要：

Milvus 有两种启动方式：MilvusLite（单机嵌入式，无需外部服务）和Milvus 服务器版（Docker 部署，适合生产环境）。开发阶段推荐 MilvusLite，生产环境推荐 Docker。

三、BGE-M3 Embedding 服务

BGE-M3 是智源研究院（BAAI）开发的多语言 Embedding 模型。它支持 100+ 种语言，同时返回稠密向量、稀疏向量和多向量三种表示，最大序列长度 8192 Token。BGE-M3 的设计目标是"一个模型解决多种检索需求"——稠密向量用于语义匹配，稀疏向量用于关键词匹配，多向量用于细粒度匹配。

使用 FlagEmbedding 库可以非常方便地调用 BGE-M3。首次运行会自动从 HuggingFace 下载模型文件（约 2.2 GB），后续运行直接从本地加载。为了节省显存，我们启用 FP16 半精度模式——在大多数 GPU 上，半精度推理的精度损失不超过 1%，但显存占用减少一半。

在实际项目中，Embedding 服务通常需要做成独立的微服务。原因有三：一是模型加载耗时长（10-30 秒），服务化后只需加载一次；二是多业务可以共享同一个 Embedding 服务，避免重复加载模型浪费资源；三是 Embedding 服务可以独立扩缩容，不与向量数据库耦合。

05_vector_embedding/08_practical_service/milvus_bge_service.py
"""
milvus_bge_service.py — Milvus + BGE-M3 向量检索服务实战
"""
import os
from typing import Optional


class BGEEmbeddingService:
    """
    BGE-M3 Embedding 服务：使用本地模型生成文本向量。

    依赖：pip install FlagEmbedding torch transformers
    """

    def __init__(self, model_name: str = "BAAI/bge-m3"):
        self.model_name = model_name
        self._model = None

    def _load_model(self):
        """延迟加载模型和分词器"""
        from FlagEmbedding import BGEM3FlagModel
        self._model = BGEM3FlagModel(
            self.model_name,
            use_fp16=True,  # 半精度推理，节省显存
        )

    def encode(self, texts: list[str]) -> dict:
        """
        将文本列表编码为向量。

        返回：
        - dict: 包含 dense_embeddings 和 sparse_embeddings
        """
        if self._model is None:
            self._load_model()

        result = self._model.encode(
            texts,
            batch_size=16,
            max_length=8192,
            return_dense=True,
            return_sparse=True,
            return_colbert_vecs=False,
        )

        return {
            "dense_embeddings": result["dense_vecs"],
            "sparse_embeddings": result["lexical_weights"],
        }

四、Milvus 向量数据库操作

Milvus 的 Python 客户端 pymilvus 提供了 MilvusClient 类，封装了集合创建、数据插入、向量检索等核心操作。相比旧版的 Connections + Collection API，MilvusClient 的使用更简洁，适合快速开发和原型验证。

Milvus 的核心概念是"集合"（Collection），相当于关系数据库中的"表"。每个集合包含多个字段，其中必须有一个主键字段和一个向量字段。还可以添加文本字段、JSON 元数据字段等。索引构建是在向量字段上创建检索结构，创建索引后才能高效搜索。

MilvusClient 的 create_schema 方法用于定义集合的数据结构，add_field 方法添加各个字段，prepare_index_params 方法配置索引参数。这种设计让集合定义和索引配置分离，更清晰也更灵活。

class MilvusVectorService:
    """Milvus 向量数据库服务"""

    def __init__(self, uri: str = "milvus_demo.db"):
        self.uri = uri
        self._client = None

    def _connect(self):
        """连接到 Milvus"""
        from pymilvus import MilvusClient
        self._client = MilvusClient(uri=self.uri)

    def create_collection(
        self, collection_name: str, dimension: int = 1024,
        auto_id: bool = True,
    ) -> None:
        """创建向量集合"""
        if self._client is None:
            self._connect()

        schema = self._client.create_schema(
            auto_id=auto_id,
            enable_dynamic_field=True,
        )
        schema.add_field("id", "INT64", is_primary=True)
        schema.add_field("vector", "FLOAT_VECTOR", dim=dimension)
        schema.add_field("text", "VARCHAR", max_length=65535)
        schema.add_field("metadata", "JSON")

        index_params = self._client.prepare_index_params()
        index_params.add_index(
            field_name="vector",
            index_type="HNSW",
            metric_type="COSINE",
            params={"M": 16, "efConstruction": 200},
        )

        self._client.create_collection(
            collection_name=collection_name,
            schema=schema,
            index_params=index_params,
        )

    def insert(self, collection_name: str, data: list[dict]) -> dict:
        """插入向量数据"""
        if self._client is None:
            self._connect()
        return self._client.insert(collection_name, data)

    def search(self, collection_name: str, query_vector: list[float],
               top_k: int = 5, filter_expr: Optional[str] = None) -> list:
        """向量检索"""
        if self._client is None:
            self._connect()
        return self._client.search(
            collection_name=collection_name,
            data=[query_vector],
            anns_field="vector",
            limit=top_k,
            filter=filter_expr,
            search_params={"metric_type": "COSINE", "params": {"ef": 64}},
            output_fields=["text", "metadata"],
        )

五、完整检索流程

将 Embedding 服务和 Milvus 服务组合起来，就形成了完整的检索服务。调用者只需要传入查询文本，服务自动完成 Embedding → 检索 → 返回结果的全流程。这种封装隐藏了底层的向量计算细节，让上层应用开发者专注于业务逻辑。

RetrievalService 的核心方法是 index_documents 和 search。前者接收文档列表，先通过 BGE-M3 生成向量，再批量写入 Milvus；后者接收查询文本，先 Embedding 生成查询向量，再到 Milvus 中搜索最相似的文档。整个过程对用户完全透明。

class RetrievalService:
    """
    完整的检索服务：整合 BGE-M3 Embedding 和 Milvus 检索。
    """

    def __init__(
        self,
        embedding_model: str = "BAAI/bge-m3",
        milvus_uri: str = "milvus_demo.db",
    ):
        self.embedder = BGEEmbeddingService(embedding_model)
        self.milvus = MilvusVectorService(milvus_uri)

    def index_documents(self, collection: str, documents: list[dict]) -> dict:
        """
        索引文档：先 Embedding，再写入 Milvus。
        """
        texts = [doc["text"] for doc in documents]
        embeddings = self.embedder.encode(texts)

        data = []
        for i, doc in enumerate(documents):
            data.append({
                "vector": embeddings["dense_embeddings"][i].tolist(),
                "text": doc["text"],
                "metadata": doc.get("metadata", {}),
            })

        return self.milvus.insert(collection, data)

    def search(self, collection: str, query: str, top_k: int = 5) -> list:
        """
        检索：先 Embedding，再搜索 Milvus。
        """
        embeddings = self.embedder.encode([query])
        query_vector = embeddings["dense_embeddings"][0].tolist()
        return self.milvus.search(collection, query_vector, top_k)

七、进阶功能

7.1 元数据过滤

Milvus 支持在检索时添加过滤表达式。比如只搜索"科技"分类的文档。过滤表达式使用类 SQL 语法，支持 ==、!=、>、<、in、like 等运算符。

# 只搜索科技分类
results = service.milvus.search(
    "rag_docs", query_vector, top_k=3,
    filter_expr="category == '科技'",
)

# 组合条件：科技分类 + 2024年
results = service.milvus.search(
    "rag_docs", query_vector, top_k=3,
    filter_expr="category == '科技' and date >= '2024-01-01'",
)

7.2 批量查询

一次可以提交多个查询向量，Milvus 会并行处理。这在需要同时处理多个用户查询或进行批量评估时非常有用。

# 批量查询
queries = ["AI 医疗", "深度学习原理", "NLP 技术"]
query_vectors = [service.embedder.encode([q])["dense_embeddings"][0].tolist()
                 for q in queries]

results = service.milvus.search("rag_docs", query_vectors, top_k=2)
# results 包含每个查询的 Top-K 结果

7.3 混合检索

BGE-M3 同时返回稠密向量和稀疏向量。可以将两者结合实现混合检索：稠密向量用于 Milvus 语义检索，稀疏向量用于 BM25 关键词匹配，两者结果融合后得到更精准的排序。

# 获取稠密和稀疏两种表示
result = service.embedder.encode(["查询文本"])
dense_vec = result["dense_embeddings"][0]
sparse_vec = result["sparse_embeddings"][0]

# 稠密向量 → Milvus 语义检索
# 稀疏向量 → BM25 关键词匹配
# 两者结果融合 → 混合检索
💡 生产环境建议
  • 模型预热：服务启动时先调用一次 encode() 加载模型到显存，避免首次请求延迟过高
  • 批量 Embedding：索引文档时尽量批量处理（batch_size=16-32），比逐条处理快 5-10 倍
  • 连接池：高并发场景下复用 MilvusClient 实例，避免频繁建连
  • 异常处理：Embedding 服务和 Milvus 都可能失败（模型加载失败、网络中断等），需要加 try-except 和重试逻辑

在实际生产环境中，还需要考虑服务的监控和告警。建议接入 Prometheus + Grafana 监控体系，跟踪 Embedding 延迟、Milvus 查询延迟、内存使用率等核心指标。设置合理的告警阈值（如 P99 延迟超过 200ms 时告警），确保问题能被及时发现和处理。

八、模块五总结

模块五从文本向量化原理出发，系统学习了 Embedding 模型的选型、向量数据库的选择、索引算法的原理、相似度度量的策略，以及混合检索和分布式部署的完整知识体系。5.8 实战将这些知识整合为一个可运行的检索服务，完成了从理论到实践的闭环。

通过本模块的学习，你应该能够：理解文本如何转换为向量，包括从 One-Hot 到上下文感知的完整演进过程；根据业务需求（语言、成本、维度、开源性）选择合适的 Embedding 模型；根据数据规模和性能要求选择合适的向量数据库和索引类型；实现向量检索、BM25 全文检索和混合检索；在生产环境中部署和优化向量数据库服务。

回顾本模块的核心知识链路：文本 → 向量化（Embedding） → 索引构建（HNSW/IVF/PQ） → 相似度计算（余弦/欧氏/点积） → 混合检索（向量 + BM25） → 分布式部署。这是一条完整的向量检索技术栈，也是构建 RAG 系统的核心基础设施。

在下一个模块六中，我们将进入 RAG（检索增强生成）系统的设计与实现，将向量检索与大语言模型结合，构建完整的智能问答系统。届时，模块五学到的所有知识——Embedding 模型选择、向量数据库部署、索引参数调优、混合检索策略——都将成为 RAG 系统的核心组件。

RAG 系统的核心流程是：用户提问 → 向量检索找到相关文档 → 将文档作为上下文传给大语言模型 → 模型生成回答。这个流程中的每一个环节都依赖模块五的知识：向量检索需要合适的 Embedding 模型和数据库，上下文拼接需要理解向量相似度的含义，而整个系统的性能优化需要掌握索引调优和分布式部署的技巧。
